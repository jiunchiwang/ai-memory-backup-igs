**#S972** Generate commit message for R-1 bugfix: drop-in animation freeze causing old reels to keep spinning during anticipation period (Jun 25, 11:26 AM)

**Investigated**: The drop-in animation lifecycle across three game paths: normal game spin, Free Game (FG) triggers, and MgOmen dormant state. Verified the fix resolves the core issue where m_isInDropMode flag was incorrectly used as idempotency key, causing promise to never be created in certain flows.

**Learned**: The bug was caused by coupling between freeze state and animation trigger state. m_isInDropMode should represent "entire drop sequence [StartSpin→drop-in] freeze state" while promise!=null indicates "drop animation has been triggered". Using m_isInDropMode as the idempotency key in TriggerDropOut prevented promise creation in FG flows (FP-1 finding), causing reels to remain spinning during the frozen anticipation period.

**Completed**: Implemented Fix Method A with 2 logic changes in SlotReels.ts: (1) StartSpin now sets m_isInDropMode=false→true (restoring first-frame freeze) and clears stale promise=null, (2) TriggerDropOut idempotency key changed from if(m_isInDropMode) to if(promise!=null), decoupling freeze flag from animation trigger state. This resolves both R-1 (high priority freeze bug) and R-2 (old reel spinning). TypeScript compilation shows 0 errors in assets/Script. Modified files: SlotReels.ts, GameView.ts, SpinState.ts, plus untracked SPEC.md.
