# ms-external-repo-absorption

當需要評估並吸收外部 repo、skill、框架、或架構文章的設計到自己的系統時使用。

## 觸發條件

- 使用者要求研究/評估外部 repo 或 skill
- 使用者分享外部框架/架構文章並要求比對借鏡
- 同事提供的 skill/工具需要整合進既有體系
- 發現外部專案有可借鏡的設計 pattern

## 步驟

### Step 0: 範圍確認

1. 確認吸收目標（repo URL / 檔案路徑 / 文章連結）
2. 確認使用者意圖：全盤移植 / 選擇性借鏡 / 純研究不吸收
3. 確認既有系統的邊界（哪些模組/功能是比對基準）

### Step 1: 現狀盤點

1. 讀取目標的核心設計（README / 架構文件 / 核心程式碼）
2. **同時盤點既有系統的對應功能**（不可跳過——直接看外部不看自己是最常見錯誤）
3. 產出一張比對表：

```
| 功能/設計 | 外部做法 | 既有做法 | 差距判定 |
|-----------|---------|---------|---------|
| ...       | ...     | ...     | 已有/借鏡/不需要 |
```

### Step 2: 差距分析 + 借鏡排序

1. 篩掉「已有」和「不需要」的項目
2. 對每個「借鏡」項目評估：
   - **增量價值**：解決什麼既有系統沒解決的問題？
   - **整合成本**：改多少檔？影響多少既有功能？
   - **風險**：有無與既有設計衝突的假設？
3. 按「價值/成本」排序，標出建議優先級

### Step 3: 方案 + 風險評估

1. 對每個要吸收的項目擬定具體方案（改哪些檔、怎麼改）
2. 逐項列風險：
   - 與既有功能的交互影響
   - 設計假設是否成立（外部的 context 不同於自己的 context）
   - 可逆性（改了能不能退回）
3. **報告使用者確認**再進入實作——不可自行決定吸收範圍

### Step 4: 實作 + 驗證

1. 按確認的範圍逐項實作
2. 每項完成後驗證（測試/type check/smoke）
3. 有衍生教訓回寫對應記錄（facts/wiki/skill 正本/pitfalls）

### Step 5: 收尾

1. 更新既有文件（AI.md / README / wiki）反映吸收結果
2. 若外部工具本體有 bug，產出回饋文件給來源方
3. 記錄本次吸收的最終狀態（哪些吸收了、哪些排除了、為什麼）

## 常見錯誤

1. **只看外部不看自己**：直接列「外部有什麼好的」卻不查既有系統是否已有同等功能，導致重複實作或誤判差距
2. **照搬不適配**：外部設計的 context（規模/架構/語言）與自己不同，直接搬過來反而製造問題
3. **吸收範圍膨脹**：研究時發現很多「順便可以做的」就全部做，scope 失控
4. **跳過使用者確認**：自行判斷吸收範圍後直接實作，事後使用者不認同
5. **忘記驗證外部的正確性**：預設外部實作是對的，未做獨立驗證就整合（實證：codegen skill 有 3 個 spec_adapter bug）
