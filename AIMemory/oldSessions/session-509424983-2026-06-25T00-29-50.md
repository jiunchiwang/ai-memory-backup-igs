## Karpathy 的 LLM Wiki 模式分析

這是 Karpathy 2026/4/4 發的 gist（5000+ stars），提出一個**用 LLM 增量維護個人知識庫**的模式。以下是核心要點：

### 核心洞察

傳統 RAG（NotebookLM、ChatGPT file uploads）每次問答都從零開始重新發現知識。LLM Wiki 不同：

> LLM **增量構建並維護一個持久化 wiki**——結構化、互相連結的 markdown 檔案。新來源不只被 index，而是被**整合**進既有知識圖——更新 entity pages、修正摘要、標記矛盾、強化或挑戰已有綜合結論。

**Wiki 是一個持續複利的 artifact。** 交叉引用已經建好、矛盾已經標記、綜合已經反映所有你讀過的東西。

### 三層架構

| 層 | 角色 | 誰寫 |
|---|---|---|
| **Raw sources** | 不可變的原始資料（論文、文章、圖片） | 人類策展 |
| **Wiki** | LLM 生成的 markdown 頁面（摘要、entity、概念、比較） | LLM 全權擁有 |
| **Schema** | 告訴 LLM wiki 結構和慣例的文件（CLAUDE.md / AGENTS.md） | 人 + LLM 共同演進 |

### 三個操作

1. **Ingest** — 新來源放進 raw，LLM 讀取 → 寫摘要 → 更新 index → 更新跨頁面的 entity/concept pages。一個來源可能觸及 10-15 個 wiki 頁面。
2. **Query** — 問問題，LLM 搜索相關頁面、合成答案。好答案可以**存回 wiki** 成為新頁面，讓探索也能複利。
3. **Lint** — 定期健康檢查：找矛盾、過時資訊、孤兒頁面、缺少交叉引用、資料缺口。

### 索引機制

- **index.md** — 內容導向的目錄，每頁一行摘要。LLM 問答時先讀 index 定位相關頁面。~100 來源 / 數百頁規模下「不需要 embedding RAG 基礎設施」。
- **log.md** — 時序 append-only 記錄（ingest / query / lint），用一致格式方便 grep。

### 工具建議

- **Obsidian** 當 IDE（graph view 看連結、Web Clipper 抓文章）
- **qmd**（BM25 + vector hybrid search + LLM rerank，本地跑）當 wiki 長大後的搜尋引擎
- Marp 做簡報、Dataview 做 frontmatter 查詢

### 為什麼能用

> 維護知識庫的困難不是閱讀或思考——是簿記。LLM 不會無聊、不會忘記更新交叉引用、可以一次改 15 個檔案。人做策展和思考，LLM 做其他所有事。
