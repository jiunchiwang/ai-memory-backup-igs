## 如果不播 Idle 動畫的影響

`PlayDownEffectSpineIdleAnimation()` 在這些時機被呼叫：

1. **`Init()`** — 初始化時立即播放
2. **`PlayLevelUpEffect()` 結束後** — 升級動畫播完切回 Idle
3. **`PlayFailedLevelUpEffect()` 結束後** — 升級失敗動畫播完切回 Idle
4. **`PlayActivateEffectEnd()` 結束後** — 發動效果結束切回 Idle

### 結論：**不播 Idle 不會有邏輯錯誤，但視覺上會有問題**

具體影響：

1. **Spine 會停在最後一幀不動** — `LVUp` / `X_Act_Out` 播完後，如果不切回 Idle，Spine 會凍結在 Out 動畫的最後一幀（可能是某個不正確的殘留畫面）。

2. **不影響後續功能運作** — 下次觸發 `PlayLevelUpEffect` / `PlayActivateEffectStart` 時會重新設定動畫，所以功能流程不會壞。

3. **Init 時不播也沒事** — 只是遊戲載入後下層效果區域不會有待機動畫而已。

### 總結

`Idle` 的角色是**視覺上的「回歸待機狀態」**，確保每次一次性動畫播完後 Spine 不會停在奇怪的幀。如果你打算移除它，要確認：
- 動畫最後一幀本身是否就是乾淨的空白狀態
- 或者改為播完後直接隱藏節點（`SetNodeActivity(false)`）

如果 Idle 動畫本身只是一個靜態 pose 或微動效果（例如微光、呼吸燈），拿掉之後使用者看到的差異就是「乘倍區域平時沒有背景小動畫了」。

### 🤖 Assistant (下午4:25:44)

是的，效能會更好。原因：

**Spine 即使播「靜止」動畫也有開銷：**
- 每幀仍在跑骨骼計算（skeleton update）
- 每幀仍在做頂點變換（mesh generation）
- 仍占一個 draw call

**換成靜態圖（Sprite）+ 隱藏 Spine：**
- 零骨骼運算
- Sprite 可被 auto-batching 合批，draw call 更低
- 記憶體差異可忽略（一張靜態圖 vs Spine atlas 本來就載了）

---

## 做法

把 `PlayDownEffectSpineIdleAnimation()` 改成：顯示靜態圖、隱藏 Spine。動畫要播時反過來。

概念上就是：

```typescript
// 原本
private PlayDownEffectSpineIdleAnimation(): void {
    if (!this.m_downEffectSpine) return;
    this.m_downEffectSpine.SetNodeActivity(true);
    this.m_downEffectSpine.SetSpineNodeActivity(true);
    this.m_downEffectSpine.PlayAnimation('Idle', this.m_downEffectSpine.Spine, true);
}

// 改成
private PlayDownEffectSpineIdleAnimation(): void {
    if (!this.m_downEffectSpine) return;
    this.m_downEffectSpine.SetSpineNodeActivity(false); // 關 Spine 運算
    // 顯示同位置的靜態圖節點
    if (this.m_downEffectIdleSprite) {
        this.m_downEffectIdleSprite.active = true;
    }
}
```

然後在 `PlayLevelUpEffect` / `PlayActivateEffectStart` 這些要播動畫的入口加上反向操作（隱藏靜態圖、開啟 Spine）。

## 實際效能差多少？

坦白說對單一個 Spine 來說差異是**微小的**（大約省 0.1~0.3ms/frame），但如果你的遊戲盤面上有多個乘倍位（例如多個 MultiplierManager instance）或搭配其他長期 idle 的 Spine，累積起來就有感。在低端手機上更明顯。
