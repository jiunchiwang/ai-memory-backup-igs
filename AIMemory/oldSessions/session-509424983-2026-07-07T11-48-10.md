## 兩邊現況

**ai_multi_agent**：`SessionStore`（record + active/archived/crashed + auto-naming + atomic write）+ ACP `session/load` 原生恢復 + `/session` inline keyboard。哲學：**保留原始 ACP context，隨時切回**。

**bridge**：`acpClient.ts:67` 明寫 `one AcpClient == one process == one session (we only use session/new once)`——從未用過 `session/load`。但 bridge 不是沒有 session 概念，而是走另一條路：
- `session-archive.ts` — 結構化歸檔（turns、goal state、close reason），恢復時**只注入摘要**（明確設計決策：避免 context 爆炸）
- `[WS]` working state + session-extract + facts 蒸餾
- 哲學：**丟掉原始 context，靠蒸餾記憶重建連續性**

## 移植價值分析

| 場景 | 現有蒸餾方案 | ACP session/load 增益 |
|------|------------|---------------------|
| bridge restart 後接續 | 摘要注入，細節丟失 | ✅ 完整 context 恢復（含工具狀態、未講完的推理） |
| idle timeout 殺 process | context 永久丟失 | ✅ **殺 process 省記憶體（~260MB），下次訊息 load 回來** —— 最強 ROI 點 |
| 平行工作流切換（任務A↔B） | 不支援 | ✅ 但單人使用需求低 |
| 長期知識連續性 | ✅ facts/wiki 已覆蓋 | ❌ 無增益 |

## 主要風險（三個坑）

1. **Replay side-effect**：`session/load` 會把歷史對話以 `session/update` 重播。bridge 的 `observerTransformer` 若在 replay 期間解析到歷史的 `<<ASK>>`/`<<SCHEDULE>>`/`<<RELAY_DELEGATE>>` token 會**重複觸發副作用**。必須加 replay 抑制旗標。這是最大的坑。
2. **與 preamble-frozen 政策的互動**：resume 回來的 session 帶著舊 preamble（舊 facts）。技術上與凍結政策一致，但 `<<RESTART>>` token 的存在意義就是「刷新 preamble」——**resume 必須是分情境 opt-in**（idle/crash → resume；RESTART token → 刻意 fresh）。
3. **Adapter 能力差異**：bridge 是手刻 JSON-RPC client（非官方 SDK），`session/load` 要自己實作（本身只是一個 RPC call，小），但 kiro-cli acp / claude-agent-acp / codex-acp 三家的 `loadSession` capability 與 replay 行為不一致，需從 `initialize` 回應偵測 + 各寫 smoke test。

## 方案選項

- **方案 A（建議）— 只做 resume，不做 UI**：persist sessionId per chat；idle/crash/非計畫 restart 時走 `session/load` 恢復，`<<RESTART>>`/`/reset` 維持 fresh。附帶解鎖「idle 殺 process 省記憶體」。改動集中 `acpClient.ts` + `sessionManager.ts`，估 200-400 行 + replay 抑制。
- **方案 B — A + SessionStore + `/session` 切換 UI**：完整照抄。多出平行工作流能力，但與 goal/MoA/relay 的單 session 假設全面互動，風險放大（scope 可縮不可膨脹原則 → 拆開做）。
- **方案 C — 不移植**：現有蒸餾方案已覆蓋 8 成需求，省下複雜度。

**我的建議**：方案 A。價值最實（restart 連續性 + 記憶體回收），風險可控（replay 抑制是唯一硬骨頭），且不動 UI 層。B 等 A 跑穩再說。若要實作，這個量級適合委派 Kiro 執行、我做規劃與 review。
