## ai_multi_agent 研究結論

### 一句話

**Python 寫的「多 Agent Forum 團隊框架」**——一個 Telegram Group 開 Forum Mode，每個 Topic 綁一個獨立 AI Agent subprocess，由中央 Daemon 管理生命週期，Agent 間透過 MCP tool + HTTP API 通訊。

---

### 核心架構

```
Telegram Group (Forum Topics)
     ↓  每個 Topic = 一個 Agent
TelegramAdapter (python-telegram-bot, topic_id → instance 路由)
     ↓
Daemon (Instance 生命週期 + Health Monitor + Session Store)
     ↓
Backend 抽象層
  ├─ ACPSdkBackend（主力，官方 ACP Python SDK）
  ├─ KiroCLIBackend（legacy stdin/stdout 文字解析）
  └─ ClaudeCodeBackend / CodexBackend（別名都走 ACPSdkBackend）
     ↕
MCP Server（team_mcp.py，stdio JSON-RPC，透過 HTTP 呼叫 Daemon API）
```

### 與你的 bridge 的關鍵差異

| 維度 | ai_multi_agent | telegram-kiro-bridge |
|------|---------------|---------------------|
| **架構模型** | 中心化 Daemon + 多獨立 Agent（各自 workspace） | 單一 Bridge + Specialist subprocess |
| **語言** | Python（框架）+ Go（獨立工具） | TypeScript (Node.js) |
| **Telegram 模式** | Forum Group（每個 Topic = 一個 Agent） | 單 Bot 私聊/群聊 |
| **Agent 通訊** | MCP tool → HTTP API → Daemon 轉發 | Text token protocol（`<<TOKEN:...>>`） |
| **Reply 機制** | Agent 呼叫 MCP `reply()` tool → HTTP → Bot 發訊 | Agent stdout streaming → Bridge 即時編輯同一則 |
| **記憶系統** | 依賴各 Agent 各自 steering/knowledge | 自建 MCP + embedding + topic shard |
| **Session 管理** | 完整支援（ACP SDK，切換/歸檔/恢復） | 以 chatId 為 key，idle timeout |
| **自我改進** | ImprovementHarness（error → Telegram topic 報告） | 14 類錯誤分類 + reflexion + capability gap |
| **動態注入** | system-overview.md + inject-rules（每次重啟重產） | Preamble 凍結 + environment block |
| **維運自動化** | `/update` 從 GitHub pull + restart | Dream pipeline（13 步每日自動） |

### 值得借鏡的設計

1. **Forum Topic 路由** — 天然的多 Agent UI 隔離，使用者在不同 Topic 跟不同角色對話，互不干擾。比單 bot 的 `/specialist` 切換更直覺。

2. **MCP 三層注入策略** — 先磁碟寫入 fallback、再 ACP `session/new` 原生注入、加啟動時序控制（先刪 team entry 防阻塞 → API ready 後才寫回）。比你的靜態 postinstall 更動態。

3. **Backend 抽象 ABC** — `start / send_message / stop / is_alive / is_ready` 五方法，換 Agent 只改 `team.yaml` 的 `backend:` 欄位。你的 bridge 也是透過 env 切換但沒有正式 ABC interface。

4. **reply() MCP tool with kind** — `primary`（每 message 只能一次）+ `followup`（補充，最多 2 次），防止 Agent 重複回覆。你用的是 streaming edit 方式不需要這個。

5. **request_knowledge** — 同步跨 Agent 知識請求（60s timeout），區別於 leader 的 `send_to_instance`（異步任務分派）。你的 Relay 是跨機，但本機 Specialist 沒有同步問答機制。

6. **Improvement Harness** — 結構化事件（error/timeout/crash/quota/fix）→ 寫 JSONL + 發到專用 Telegram Topic，清楚區隔「維運通知」和「使用者對話」。你的事件分散在 event-log + hit-log。

7. **inject-rules 外部化** — `all.md`（所有 agent）+ `leader.md`（leader only）+ `worker.md`（worker only），讓行為規範跟 code 解耦。你用的是 preamble 硬寫。

8. **team.yaml 一站式配置** — 一個 YAML 定義所有 instance、backend、model、role、topic_id、allowed_users。你的分散在 .env + specialist-domains.json + relay-peers.json。

### 相對弱項（你的 bridge 領先的地方）

- **沒有 streaming edit** — Agent 回覆走 MCP `reply()` 一次性送出，沒有即時 chunk 編輯
- **沒有 embedding / 語意路由** — 沒有本地 ONNX、沒有 memory recall / skill routing
- **沒有 text token protocol** — 不支援 `<<SEND_FILE>>` `<<ASK>>` `<<SCHEDULE>>` 等嵌入式指令
- **沒有 goal 系統** — 沒有多 turn 自動執行的持續性目標
- **沒有本地 LLM 加速** — 沒有 llama.cpp 整合
- **沒有跨機 relay** — 只能同機器，沒有 bot-to-bot 跨機協作
- **沒有 Dream pipeline** — 維運靠手動 `/update`

### 技術細節補充

- Python 3.11+，使用 `from __future__ import annotations` + dataclass
- ACP Python SDK：`agent-client-protocol` package（typed models，非手寫 JSON-RPC）
- Daemon 有 Health Monitor（30s 間隔）+ Auto-Restart（指數退避，max 10 次）
- Warm Pool：預熱 subprocess 池（減少冷啟動延遲）
- 開機自啟：Windows Task Scheduler + `autostart.bat` + `vbs` 隱藏視窗
- Git flow：`master`（開發）+ `release`（穩定），tag 格式 `YYMMDDNN`

### 方案 1：ACP Protocol Trace（推薦 P0）

**他們怎麼做**：`acp_tracer.py`（~75 行），每個 session 寫一個 `acp-trace-<hash>.jsonl`，記錄每條 send/recv 的 JSON-RPC + timestamp。API endpoint `/api/trace` 可讀最近 50 條。

**我們為什麼需要**：目前 debug ACP 問題（如 session/update 異常、tool call 卡住）只能靠 console log 猜。有了 trace 就能回放整段協議互動。

**如何吸收**：在 `acpClient.ts` 的 `_sendRpc` / `_handleLine` 各加一行 append JSONL，存到 `${MEMORY_DIR}/acp-trace/`。不影響現有 event-log（event-log 記的是高層事件，trace 記的是原始協議）。

**不衝突原因**：跟 PostTool Hooks / Loop Budget / Karpathy P0 完全正交。

---

### 方案 2：Session 歸檔/恢復（推薦 P1）

**他們怎麼做**：`session_store.py` 用 JSON atomic write 追蹤每個 session（id, name, status, prompt_count, first_prompt, auto-naming）。Telegram `/session` 指令用 inline keyboard 讓使用者切換 active sessions 或恢復 archived。

**我們為什麼需要**：目前 session idle 30min 就砍掉，如果 context 還有價值（例如研究到一半被打斷），就只能靠 /save + 手動 handoff。有了歸檔機制可以「暫放」再回來。

**如何吸收**：`sessionManager.ts` 加 `SessionRecord` 結構 + JSON 持久層，idle timeout 改為「歸檔」而非「銷毀」。ACP session/load 恢復。UI 用 ASK button 列出可恢復的 session。

**不衝突原因**：跟 Karpathy P0 的 Query Auto-save 是不同層——一個是 session 本身的生命週期，一個是知識萃取。

### 方案 3：Reply Dedup（推薦 P0）

**他們怎麼做**：`api.py` 對每個 `(instance, hash(message))` 做 5 分鐘去重。`daemon.py` 追蹤 `_primary_replied` set，確保每個 user message 只有一次 primary reply。

**我們為什麼需要**：Specialist delegate 偶爾會因 retry 或 continuation 造成重複訊息注入主 session。Goal auto-continuation 也可能撞到。

**如何吸收**：在 `specialist.ts` 的 result injection point 加 hash dedup（用 message hash + 5min window）。~40 行，純防禦性，不改現有 flow。

**不衝突原因**：純 bug prevention，不碰任何 roadmap 項目。

### 方案 4：Conversation Summarizer（P2）

**他們怎麼做**：`conversation_summarizer.py` 在 restart 前讀 Kiro CLI SQLite 的最後 N 輪對話，spawn 一個臨時 ACP session 跑摘要 prompt，回傳 3-5 句結論。

**我們為什麼需要**：目前 `/restart` 時 working state 只存 task/next_action 這些結構化欄位。如果能自動跑一輪「這個 session 討論了什麼」，下個 session 恢復得更快。

**如何吸收**：在 `sessionManager.ts` 的 `closeSession` 前，用已有的 session transcript 跑一輪輕量 summarize prompt（可用 local LLM 省成本），結果存到 working state 的 `summary` 欄位。

### 方案 5：Runtime Health Monitor（P1）

**他們怎麼做**：Daemon 每 30s 跑一輪 health check（`is_alive()`），崩潰時指數退避 auto-restart（最多 10 次），連續失敗觸發 `record_crash()` 到 ImprovementHarness。

**我們為什麼需要**：目前只有啟動時 `runHealthCheck()`。如果 ACP subprocess 中途死掉（OOM、被防毒砍、pipe 斷），bridge 要等到下一次 user message 才會發現。

**不衝突原因**：跟所有 roadmap 項目正交，純可靠性提升。
