psmux的pane是什麼?

---

### 🤖 Assistant (上午11:04:15)

psmux 的 pane 就是 tmux 的 pane 概念——**一個 terminal 視窗內的分割區塊**，每個 pane 跑獨立的 shell/程序。

簡單說：

- **Window** = 一整個全螢幕 terminal 畫面
- **Pane** = 把一個 window 切成多塊（水平/垂直分割），每塊各跑自己的程式

```
┌────────────────────────────────┐
│  pane 0: bridge (npm run dev)  │
├────────────────────────────────┤
│  pane 1: llama.cpp server      │
├────────────────────────────────┤
│  pane 2: tail -f event-log     │
└────────────────────────────────┘
```

操作方式和 tmux 一樣（psmux 講 tmux command language）：

- `Ctrl+B %` — 垂直分割
- `Ctrl+B "` — 水平分割
- `Ctrl+B ↑↓←→` — 切換焦點 pane
- `Ctrl+B z` — 放大/縮小單一 pane

每個 pane 有獨立的 scrollback、獨立的 process，互不影響。關掉 terminal 再 attach 回來，所有 pane 的狀態都還在。

幫我查一下github倉庫有哪些方式可以從a組織移轉至b組織?

---

### 🤖 Assistant (下午1:53:32)

GitHub repo 從 A 組織移轉到 B 組織，主要有 **3 種方式**：

---

## 1. Transfer Repository（官方內建移轉）

最正規的方式。Settings → Transfer ownership → 輸入目標組織名。

**保留的東西：**
- Git history（完整）
- Issues、PRs、Releases、Projects
- Stars、Watchers
- GitHub Pages 設定
- 舊 URL 自動 301 redirect 到新位置

**條件：**
- 你必須是源 repo 的 admin
- 你必須有目標組織的 repo 建立權限
- ⚠️ Internal repo 不能跨 Enterprise 移轉（同一個 Enterprise 內的組織才行）
- Webhook / Deploy key / Branch protection 會保留，但 **team 權限需重新設定**

**適合：** 同 Enterprise 下的組織間搬家、一次搬一個 repo。

---

## 2. GitHub Enterprise Importer（GEI）

用 `gh gei` CLI 或 GraphQL API 做批量遷移。

```bash
gh gei migrate-repo \
  --github-source-org OrgA \
  --source-repo my-repo \
  --github-target-org OrgB \
  --target-repo my-repo
```

**保留的東西：**
- Git history、branches、tags
- Issues、PRs（含 comments）
- 可選遷移 Actions workflows

**適合：** 跨 Enterprise 遷移、批量遷移多個 repo、需要 audit log。

---

## 3. Mirror / Fork + Push（手動）

不走官方移轉，自己搬：

```bash
# bare clone 保完整歷史
git clone --mirror https://github.com/OrgA/repo.git
cd repo.git

# 在 OrgB 建空 repo，然後 push
git remote set-url origin https://github.com/OrgB/repo.git
git push --mirror
```

**保留：** 只有 Git history（branches、tags、commits）
**不保留：** Issues、PRs、Stars、Webhook、Pages — 全部丟失

**適合：** 只需要 code 不需要 issue/PR 歷史、跨平台（GitHub→GitLab）、或你沒有 transfer 權限時的 workaround。

---

## 比較表

| | Transfer | GEI | Mirror |
|---|---|---|---|
| Issues/PRs | ✅ | ✅ | ❌ |
| URL redirect | ✅ 自動 | ❌ | ❌ |
| 跨 Enterprise | ❌ | ✅ | ✅ |
| 批量操作 | ❌ 逐個 | ✅ | ✅（script） |
| 權限需求 | 雙邊 admin | PAT + org owner | 目標 org 建 repo 權 |
| Team 權限 | 需重設 | 需重設 | 需重設 |

**步驟：**
1. 進 repo → Settings → 最下方 Danger Zone → **Transfer**

4. 如果目標組織需要確認，org owner 會收到通知

**注意事項：**
- 舊 URL（`github.com/OrgA/repo`）自動 301 redirect，現有的 git remote、連結、CI 不會馬上斷
- Team 權限不會自動帶過去，搬完後要在 OrgB 重新設 team access
- GitHub Actions workflow 裡如果有 hardcode `OrgA` 的 path 要手動改
- Webhook 保留，但如果 webhook URL 含組織名要檢查
- Fork 關係會斷開
