## 這張圖的核心主張 vs 現實

### 優點（值得借鏡的）

**1. 異質模型互審是真正有價值的概念**

同一個模型自我 review 有 self-bias——它傾向認同自己的推理路徑。用不同家的模型做 reviewer（Gemini 審 Codex 的產出），理論上能抓到「同溫層盲點」。你的 bridge 目前 specialist 全用 Claude 家族（opus/sonnet），審核者和執行者的認知偏差可能重疊。

**2. Orchestrator 只佔 10% token 的設計**

把昂貴模型（Fable 5 = 推測是 Claude 最強級）限縮在「切工單 + 最終驗收」，bulk work 丟給便宜模型。這跟你已經在做的事一致（opus 主 agent 做決策，sonnet specialist 做實作），但他們更極端——orchestrator 幾乎不碰實作 context。

**3. 「送代收斂」迴圈有明確退出條件**

不是無限 retry，而是每輪都有驗收標準比對。你的 goal system 有 turn budget，但驗收標準是 agent 自己判斷 GOAL_DONE，沒有獨立第三方驗證。

---

### 疑點 / 弱點

**1. 數字太漂亮，缺乏可驗證性**

「92% 完成率 vs 傳統 35%」「Bug -82%」——這些數字沒有定義基準線。什麼算「完成」？什麼類型的任務？三個「專案實戰」沒有 repo 連結、沒有可重現的 benchmark。宣傳性質重。

**2. 延遲和協調成本被低估**

三模型來回 = 至少 3x round-trip。一個簡單 bug fix 要走完 5 步？你 bridge 的 specialist delegation 已經體會到——parallel delegate 的結果注入回主 agent 時，context 膨脹很快。三模型協調的 orchestration overhead 在小任務上可能得不償失。

**3. 「Fable 5 Token 超級省」是 misleading**

省的是 Fable 5 的 token，但 Codex 5.5 佔 60%——如果 Codex 5.5 是 OpenAI 的 API，那成本不見得比單用 Claude 便宜。「總省 65%」的基準線是什麼？跟「全用 Opus 4.8」比？那當然省，但跟「全用 Sonnet」比呢？

**4. 缺乏錯誤恢復機制**

圖上沒提到：如果 Gemini 和 Codex 意見矛盾怎麼辦？如果 Fable 5 的工單切錯了怎麼回溯？你 bridge 有 reflexion policy + tool fail tracking + goal reflection retry，這些才是實戰中真正需要的。

### 如果要從中借一個概念

我覺得最值得的是：**在 specialist 完成後加一個異質 reviewer 步驟**。

具體做法：specialist（sonnet）產出 artifact 後，用 trio-llm（你已有 Gemini Flash fallback）做一輪 quick sanity check，不是完整 review，而是 3 個 yes/no 問題：

1. 有沒有明顯遺漏需求？
2. 有沒有邏輯矛盾？
3. 跟現有架構有沒有衝突？

只有三題都 pass 才注入主 agent，否則標記讓主 agent 知道需要人工看。成本很低（Gemini Flash），但能抓到一部分 self-bias。
