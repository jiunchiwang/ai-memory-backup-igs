Roadmap 條目本身很簡短，查 wiki 來源補脈絡。## Loop Budget 顯式化（roadmap 3a）

**來源**：吸收 `cobusgreyling/loop-engineering`（⭐2.7k）時借鏡的四條操作紀律之一（L1 先行、**顯式 budget**、三段式 state、attempt cap）。當時判定 bridge 核心能力已超越該專案，只值得抄紀律面。

**要解的問題**：bridge 有多種「沒有人在 loop 裡逐 turn 把關」的自主消耗路徑——

- `/goal` continuation（自動排下一輪）
- `<<PARALLEL_DELEGATE>>` / `<<RELAY_DELEGATE>>`（一個 turn spawn 多個 specialist instance）
- `<<SCHEDULE>>` 排程、每日 `/dream` 維運

這些 loop 的 token 成本目前是**隱式的**：單點有護欄（RESTART 每 goal 上限 5 次、attempt cap、context budget discipline），但**沒有跨 loop 的總量帳本**。如果某個 goal 卡在 continuation 迴圈、或 parallel delegate 開太兇，你是事後從帳單發現，不是從系統看到。

**具體要做的**（roadmap 描述）：
1. **每日 token aggregate** — 把各自主路徑的消耗彙總成日帳
2. **趨勢報告** — 看得出「這週 delegate 用量翻倍了」這類異常斜率
3. 目的：防 loop 爆 cost，把「隱式消耗」變成「顯式可觀測」

**跟現有 Context Budget discipline 的差別**：現有的是 *per-task* 事前紀律（研究前估量、>10 tool calls 就委派）；Loop Budget 是 *fleet-level* 事後帳本——單任務守規矩不代表總量不失控。

**為什麼卡前置「等 Parallel Delegate 大量使用」**：目前自主 loop 用量還低，帳本基礎設施的建置成本 > 失控風險。等 parallel delegate 成為日常用法、多 instance 並行成本真的有爆的可能，才值得動工——這也正是 loop-engineering 自己的 L1 先行原則（先人工觀察，量大了才自動化）。
