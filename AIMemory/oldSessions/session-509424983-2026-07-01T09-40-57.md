## sample-aidlc-kiro-power 研究摘要

**是什麼**：AWS 官方範例的 Kiro Power（MIT-0 授權），實作一套「AI-DLC（AI-Driven Development Life Cycle）」自適應開發方法論。

**定位**：給 Kiro IDE（GUI 版）用的 Power plugin，不需要 MCP、不需要外部依賴，純靠 steering + workflow markdown 檔案驅動 agent 行為。

---

### 結構

```
powers/ai-dlc-methodology/
├── POWER.md              ← Power 的 manifest + 全域說明
├── steering/
│   └── core-workflow.md  ← 核心指令（ALWAYS loaded by Kiro steering system）
└── workflows/
    ├── common/           ← 9 個通用規則檔（session 復原、問題格式、歡迎訊息…）
    ├── inception/        ← 7 個 Inception 階段步驟檔
    ├── construction/     ← 6 個 Construction 階段步驟檔
    └── operations/       ← placeholder
```

---

### 方法論核心：三階段自適應

| 階段 | 目的 | 必做 | 條件式 |
|------|------|------|--------|
| 🔵 Inception | 決定 WHAT/WHY | Workspace Detection → Requirements Analysis → Workflow Planning | Reverse Engineering、User Stories、Application Design、Units Generation |
| 🟢 Construction | 決定 HOW | Code Generation（per-unit）→ Build & Test | Functional Design、NFR Requirements、NFR Design、Infrastructure Design |
| 🟡 Operations | DEPLOY/RUN | — | placeholder（未來擴充） |

**自適應邏輯**：根據使用者意圖清晰度、既有 codebase 狀態、複雜度、風險自動跳過不必要的階段。簡單 bug fix 只跑 3 個必做；greenfield 全 pipeline 走完。

---

### 機制特色

1. **Adaptive Depth**：Requirements Analysis 分 minimal / standard / comprehensive 三級
2. **Per-Unit Loop**：大專案拆 units，每個 unit 獨立走 design→code→test
3. **Explicit Approval Gates**：每個主要階段完成後等使用者確認才進下一步
4. **Audit Trail**：所有使用者輸入逐字記錄到 `audit.md`（ISO 8601 時間戳）
5. **Plan-Level Checkbox**：用 markdown checkbox 追蹤每一步進度
6. **Content Validation**：Mermaid diagram 等複雜內容必須先驗證再寫檔
7. **Session Continuity**：`aidlc-state.md` 記工作狀態，中斷可恢復
8. **2-Option Standardized Responses**：Construction 階段統一只問「Request Changes / Continue」，禁止 emergent 3-option 行為

### 可借鏡的點

1. **Adaptive depth 自動降級**——你的三級靠使用者/agent 人工判斷；AI-DLC 的多因子分析（intent clarity + codebase state + complexity + risk）可以融入你的 steering，讓 agent 自己決定走精簡或完整閉環。
2. **aidlc-state.md 單一狀態追蹤檔**——類似你的 `STATE.md` 三段式，但它把 per-stage checkbox 進度也放進去，跨 session 恢復更精確。
3. **Plan-Level Checkbox 雙層追蹤**——比你的 todo_list 更細粒度，值得在 SPEC.md 裡試用。
4. **Overconfidence prevention（獨立檔案）**——你的 Reflexion policy 偏重「失敗後修正」，AI-DLC 多了「事前防止過度自信」的 steering，可當一段 preamble 注入。
5. **Per-Unit 獨立閉環**——大型 refactor 不是一次做完全部，而是每個 unit 走完整 design→code→test 再做下一個，降低 context 爆炸風險。

---

### 不適合直接搬的部分

- **execution model 不同**：AI-DLC 是 Kiro IDE 的 Power（GUI、readFile 動態載入 workflow md），你的環境是 CLI preamble injection + Telegram streaming，不需要它的 welcome-message 儀式和 POWER.md manifest。
- **Approval gates 過多**：Telegram bot 場景使用者打字成本高，每 stage 都 explicit approval 會很煩；你的 ASK button + goal auto-continuation 已經解決了互動成本問題。
- **audit.md 逐字記錄**：你已有 event-log.jsonl + session transcript + hit-log.jsonl 做追蹤，不需要再加一份文字 audit。
- **零 MCP**：這是它的限制不是優點——因為是 sample project 所以刻意不依賴外部，你的 memory MCP + specialist 已經超越它的能力範圍。

---

**結論**：這是 AWS 官方示範「怎麼用 Kiro Power 系統打包一套方法論」的 reference implementation。方法論本身是中規中矩的 adaptive SDLC，最大價值是 Power 的封裝結構（`POWER.md` manifest + `steering/` + `workflows/` 按階段拆檔的 convention）。跟你既有的 closed-loop + karpathy-guardrails 相比，概念高度重疊但實作路線不同——你的 Telegram bridge 環境已經用更進階的機制（語意路由、reflexion、local LLM、specialist delegation）解決了同樣的問題。
