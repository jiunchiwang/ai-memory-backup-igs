gh CLI 沒登入，改用 WebFetch 直接抓頁面內容。## AgEnD Terminal 摘要

**定位**：Rust 寫的 AI coding agent 編排工具——用一個 `fleet.yaml` 宣告整支 AI 開發團隊，每個 agent 跑成獨立 PTY 長駐進程，系統自動協調。

**核心設計**：
- **Fleet-as-code**：YAML 集中配置每個 agent 的 backend/角色/工作目錄
- **5 種 backend**：Claude Code、Codex、Kiro、OpenCode、Antigravity CLI
- **29 個 MCP 協調工具**：agent 之間 send/inbox/task 互通，做真正的 swarm 協作
- **Git worktree 自動隔離**：每個 agent 各自 worktree，避免互相踩檔案
- **多通道控制**：TUI + Telegram + Discord
- **崩潰恢復**：自動 context 移交

**成熟度**：20 星、pre-alpha 但核心功能 beta 穩定、v0.10.0（2026-07）、2846 commits、11 open issues，算活躍維護中的小專案。

### 跟 bridge 的對比

| | agend-terminal | telegram-kiro-bridge |
|---|---|---|
| 定位 | 多 agent swarm 編排（agent 之間互通） | 單一人類↔單一 agent 的聊天橋接（+specialist 委派）|
| Backend 切換 | fleet.yaml 靜態宣告多個同時跑 | `.env` 切一個 ACP backend |
| Agent 協調 | 29 個 MCP tool 做 send/inbox/task | RELAY_DELEGATE / PARALLEL_DELEGATE（單向委派，非雙向 inbox）|
| 隔離機制 | Git worktree 自動隔離 | specialist 各自固定工作目錄，非 per-task worktree |

### 🤖 Assistant (下午5:09:38)

可以，而且是有明確做過 Windows 平台適配的，不是只靠 Rust 理論跨平台就算數。

證據在 `Cargo.toml`：
- **PTY 層**用 `portable-pty`（跨平台虛擬終端庫，本身就抽象掉 Unix pty vs Windows ConPTY 差異）
- 有明確的 `[target.'cfg(windows)'.dependencies]` 區塊：`windows-sys`（含 Threading / Console / Registry / JobObjects）+ `junction`（用 junction point 做目錄連結，特意避開需要系統管理員權限的作法）
- 還有 `[target.'cfg(windows)'.build-dependencies]` 的 `embed-resource`（Windows 專用的建置期資源嵌入，通常是塞 icon/manifest）

這代表開發者確實花心思處理了 Windows 專屬機制（例如 job object 管理子進程、避開權限提升需求），而不是「反正 Cargo 能跨平台編譯就算了」。README/KNOWN_ISSUES 沒明講「支援 Windows」這句話，但程式碼層級的依賴結構顯示是有備而來的。

沒查到任何 Windows 相關的已知問題或 issue——但這也可能只是專案還小（20 星、11 個 open issue）沒人在 Windows 上大量踩過，不是「保證沒 bug」的意思。
