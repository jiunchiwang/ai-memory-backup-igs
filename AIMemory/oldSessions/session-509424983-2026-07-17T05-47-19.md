### 🤖 Assistant (上午11:01:49)

Turn-lint 是一個**自動檢查機制**，在每次 AI 回覆送出前掃一遍內容，抓兩種常見違規：

1. **問句沒附按鈕** — 回覆裡問了「要不要做 X？」但沒有附 ASK 選項按鈕讓你點
2. **語言不一致** — 整段都是中文，結尾卻突然冒出一句英文

起因是 Claude Code 在多個 session 反覆犯這兩個錯（被糾正 4 次以上還犯），人工糾正治標不治本，所以直接寫一個程式層的檢查器——不管哪個 model 在回覆，都會過這關。

目前它只是 `console.warn`（印警告到 log），不會擋住訊息或自動改寫。定位是「診斷儀表板」而非「強制攔截器」。

### 🤖 Assistant (上午11:03:32)

這是 Claude Code Insights 報告裡提的一個**未來展望**構想，拆成兩個部分：

**1. 升級為 pre-response hook**

目前 turn-lint 只是印 warning 到 console log（事後診斷），不會攔截或修改回覆。升級的意思是把它變成**送出前的攔截器**——AI 回覆在抵達你之前，先過 turn-lint 檢查，不合格的直接打回去重寫，你永遠看不到違規內容。

但有個現實限制：bridge 走 ACP 路徑時，`tool-hooks.ts` 的 PostToolUse hooks 不會 fire（只有 DirectProvider 才生效）。所以真要做攔截，得在 `run-prompt.ts` 的最終渲染層動手，而不是靠 Claude Code 原生的 hook 機制。

**2. Fable5 audit**

指用 Claude Fable 5 模型做**獨立第二意見審查**——把回覆丟給另一個 model instance 判斷「有沒有違反繁中規則 / 漏 ASK button」。概念類似你之前實測的 `claude -p --model fable "prompt"` 輕量呼叫。

兩者合起來的願景：

```
AI 產出回覆
  → turn-lint 機械檢查（快、零成本、抓格式問題）
  → Fable5 語意審查（慢一點、抓邏輯層違規）
  → 都過了才送到 Telegram
```

**我的看法**：機械檢查升級成攔截是可行且值得的（延遲幾乎為零）；Fable5 audit 每 turn 都跑的話會加延遲和成本，比較適合抽樣或只在高風險 turn（如 summary / 收尾）才觸發。
