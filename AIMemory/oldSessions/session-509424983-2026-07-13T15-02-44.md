## 設計核心：勾選制量化評分 + 跨 backend token + bridge 端覆核

**1. 六維度評分（不准模型直接寫總分，只能從勾選算出）**

| 維度 | 配分 | 判定方式（需附證據） |
|---|---|---|
| V 型別驗證 | 25 | `npx tsc --noEmit` 0 error（貼輸出尾行） |
| T 功能驗證 | 20 | 對應 `check-*.mjs` 或手動驗證路徑 |
| I 呼叫端影響 | 20 | grep 呼叫者逐一列「檔:行—影響判定」（=0 需說明） |
| S 範圍紀律 | 15 | 實際改動檔案 ⊆ 事前宣告範圍 |
| C 完成度 | 10 | 無新增 TODO/stub（貼 grep 結果） |
| R 回讀驗證 | 10 | 每個編輯點改後有 read/grep 覆核 |

N/A 超過 2 項 → 總分封頂 70（防「全 N/A 卻拿滿分」）。閾值 **≥80 通過**；60-79 只重做未過項（上限 2 輪）；<60 或重做仍未過 → 併入既有 Section 4 熔斷（不新開一套計數器）。

**2. 跨 backend 落地：新 token，仿現有 `<<SKILL_USED>>` 先例**

```
<<SELF_EVAL:total|V:25,T:20,I:20,S:15,C:10,R:10|一句話證據摘要>>
```

這是純文字指令寫進 preamble，天生跨 Kiro CLI / Codex / Claude 三個 backend——因為它不依賴任何 CLI 專屬工具，只是模型輸出文字，`src/observerTransformer.ts` 用正則抓（跟 `SKILL_USED_RE` 同一套模式）。

**3. 唯一真正 model-independent 的一步（也是這方案最值錢的部分）**：bridge 端收到 token 後**自己跑 `tsc --noEmit` 覆核 V 項**，模型宣稱過但實跑失敗就標記 discrepancy 並記錄。另外「本輪有 fs 修改但沒吐 SELF_EVAL token」時 bridge 主動補提醒——這是唯一能做到「近似強制」的點，其餘全靠模型自律。

**4. 誠實分級，不裝三個 backend 保證等同**
- L1（Kiro CLI / Codex）：文字自評 token + bridge tsc 覆核
- L2（Claude Agent）：L1 之上疊 `/dev-review` 異源 skeptic 重算分數，差距 >15 分以低分為準

token 帶 L1/L2 標記讓你知道這次評分的可信等級，不宣稱三者一樣可靠。
