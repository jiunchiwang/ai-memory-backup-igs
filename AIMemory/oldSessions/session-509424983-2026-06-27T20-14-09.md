## 核心差異

| 維度 | Claude Code Hooks | 我們的方案（A→B→C） |
|------|------------------|-------------------|
| **觸發點** | 靜態配置檔（`.claude/settings.json`），寫死 `PostToolUse` matcher（檔名 glob / tool name） | 方案 B 是 runtime registry + pattern matcher，可動態增刪 |
| **執行環境** | 本機 shell script（bash/Python），Claude Code 自己 spawn | Bridge 內 JS callback，跑在 Node process 裡 |
| **感知範圍** | 只看到單次 tool call 的 input/output | 方案 A 已有全量 tool_call event log，B/C 可回溯歷史模式 |
| **學習能力** | ❌ 無 — hook 是人寫死的，不會自己演化 | 方案 C（Instinct Observer）從 tool 使用模式自動萃取 instinct → confidence scoring → SKILL_PROPOSE |
| **目的** | Lint / format / test guard（品質閘門） | 品質閘門 **+** 自我改進（把反覆操作固化成 skill 或 script） |
| **跨 agent** | 僅 Claude Code 本地 CLI | 方案 B/C 跨所有 ACP agent（Kiro/Codex/Claude/Gemini 都吃同一套） |

---

## 用一句話說

Claude Code 的 Hook 是**靜態規則觸發外部 script**（相當於 git hook 的翻版）；我們的方案是**動態 event stream → pattern recognition → 自動提議 skill/script**，從 A（被動記錄）到 C（主動學習）是一條連續光譜。

Claude Code 解決的問題：「tool 跑完後自動跑 lint」。
我們要解決的問題：「agent 反覆用笨方法做同一件事時，自動發現並固化成更好的工具」。

兩者不衝突——Claude Code 的 hook 是我們方案 B 的一個子集（matcher + callback），但我們多了 event log 回溯和方案 C 的自動萃取層。
