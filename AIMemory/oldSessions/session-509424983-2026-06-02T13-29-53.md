## Claude Code Dynamic Workflows（2026-05-28 Research Preview）

### 核心概念

Dynamic Workflow 是一段 **JavaScript 編排腳本**，由 Claude 針對你的任務即時生成，然後交由 runtime 在背景執行。它能同時調度**幾十到上千個 subagent**（上限 1,000 agents/run、並行上限 16），中間結果存在 script 變數裡而不佔 context window，你的 session 保持可用。

### 觸發方式

| 方式 | 說明 |
|------|------|
| **prompt 裡加 `workflow`** | Claude 偵測到關鍵字就自動寫編排腳本 |
| **`/effort ultracode`** | 整個 session 都讓 Claude 自己決定要不要開 workflow（xhigh reasoning + 自動編排） |
| **`/deep-research <question>`** | 內建 bundled workflow，多角度搜尋 + 交叉驗證 + 出報告 |
| **跑已存的 workflow** | `/workflows` → save 過的可當 slash command 重跑 |

### 與其他多 agent 機制的比較

| | Subagents | Skills | Agent Teams | **Workflows** |
|--|-----------|--------|-------------|--------------|
| 誰決定下一步 | Claude turn-by-turn | Claude 照 prompt | Lead agent | **Script** |
| 中間結果存哪 | context window | context window | shared task list | **script 變數** |
| 可重複性 | worker definition | instructions | team definition | **編排腳本本身** |
| 規模 | 幾個/turn | 同 subagents | 幾個長跑 peer | **幾十~幾百/run** |

### 主要優勢

1. **把計畫變成 code** — 不是 Claude turn-by-turn 決定，而是 JS 腳本控制 loop / branch / 變數
2. **內建對抗式驗證** — 可以讓獨立 agent 互相 review，錯誤在匯報前就被過濾
3. **可暫停 / 恢復** — 同 session 內 pause 後 resume，已完成的 agent 用 cache
4. **可存檔重跑** — save 到 `.claude/workflows/` 或 `~/.claude/workflows/`，變成 slash command
5. **接受 args** — 存檔後的 workflow 可傳參數（路徑列表、問題字串等）

### 限制

- 跑起來不接受 mid-run user input（只有 permission prompt 會暫停）
- Workflow 腳本本身不能直接讀寫檔案或跑 shell — 那是 subagent 的事
- 並行上限 16 agents（受機器 CPU 限制）
- 單 run 最多 1,000 agents
- token 消耗遠大於一般 session，建議先小範圍試跑
- Research Preview 階段，需要 v2.1.154+

### 實際案例

- **Bun runtime Zig→Rust 移植**：75 萬行 Rust，11 天完成，99.8% test 通過。數百 agent 並行寫 .rs + 2 個 reviewer/檔
- Codebase-wide bug hunt / security audit
- 大規模 migration（framework swap、API deprecation、跨語言 port）
- 需要多角度交叉驗證的決策（adversarial review）

### 管理方式

- `/workflows` — 列出 running/completed workflows，可 drill-in 看各 phase 的 agent 狀態
- 快捷鍵：`p` 暫停/恢復、`x` 停止、`s` 存檔、`r` 重啟單 agent
- 關閉：`/config` toggle 或 `~/.claude/settings.json` 設 `"disableWorkflows": true`

### 編排層（Orchestrator）

| | **你的 Bridge** | **Claude Code Workflows** |
|--|----------------|--------------------------|
| 編排者是誰 | Bridge TypeScript code（硬寫的邏輯） | Claude 即時生成的 JS 腳本 |

| 可重用性 | dream.json 設定步驟順序 | workflow 存檔變 slash command，可帶 args |
| 彈性 | 改流程要改 code 或 JSON | 每次任務 Claude 重寫腳本，完全動態 |

### 並行執行

| | **你的 Bridge** | **Claude Code Workflows** |
|--|----------------|--------------------------|
| 並行機制 | `<<PARALLEL_DELEGATE:task_id\|specialist\|goal>>` 同時 spawn 多個 AcpClient | Runtime 最多 16 concurrent agents（cap 1,000/run） |
| 實際並行數 | 取決於你定義了幾個 specialist（目前 4 個） | 單 run 可達數百個，每個是獨立 Claude session |
| Agent 隔離 | 每個 specialist 有獨立 skill subset + preamble + cwd | 每個 subagent 是 isolated session，共用 tool allowlist |
| 中間結果 | Artifact JSON 存到 `artifacts/`，結果注入回主 agent context | 存在 script 變數裡，不進任何 agent 的 context |

### 品質保證

| | **你的 Bridge** | **Claude Code Workflows** |
|--|----------------|--------------------------|
| 驗證方式 | Goal reflection retry（連續失敗 ≥2 自動反思）、tool fail tracking | 腳本內建 adversarial review — 獨立 agent 互相打槍 |
| 交叉檢查 | 沒有（各 specialist 獨立回報） | 原生支援多角度 draft → 投票 → 過濾不通過的 |
| 錯誤恢復 | Reflexion policy + error classifier 14 類 | 暫停/恢復，已完成的 agent 用 cache 不重跑 |

### 通訊架構

| | **你的 Bridge** | **Claude Code Workflows** |
|--|----------------|--------------------------|
| 協定 | ACP JSON-RPC over stdio | Claude Code 內部 runtime（不走 ACP） |
| 跨機器 | ✅ Relay bot-to-bot + Redis | ❌ 單機，subagent 都在本地 |
| 對外暴露 | Telegram bot = UI 層 | CLI / Desktop / VS Code / API |
| MCP 整合 | ✅ Agent 各自掛 MCP server | ✅ Subagent 繼承 parent 的 tool allowlist |

### 根本差異

**你的 Bridge 是「外部 orchestrator」模式** — Bridge code 決定什麼時候 spawn、怎麼 route、結果怎麼注入。Agent 是被動的 worker。

**Claude Code Workflow 是「agent 自編排」模式** — Claude 自己寫 orchestration script，runtime 只是執行器。計畫本身是 agent 的產出。

### 你的架構優勢

1. **跨 agent 通用** — Kiro / Codex / Claude / Gemini 都能接，不綁死一家
2. **跨機器** — Relay 讓不同硬體能力的 agent 協作，Workflow 做不到
3. **持久記憶** — MCP memory + topic shard + wiki，workflow 跑完就沒了
4. **使用者互動** — ASK button 讓人類隨時介入；Workflow 跑起來不接 user input
5. **成本控制** — specialist 用同一個 ACP session 不額外開；Workflow 每個 subagent 都是獨立 session 燒 token

### Claude Code Workflow 的優勢

1. **動態計畫** — 不用事先定義流程，Claude 看任務自己設計編排
2. **規模感** — 單 run 1,000 agents，適合「掃整個 repo」這種暴力活
3. **內建交叉驗證** — adversarial review pattern 是 first-class
4. **腳本可讀可改** — 你可以看到 JS 腳本、手改、rerun
5. **context 不膨脹** — 中間結果不進 context，只有最終答案回來

### 可借鏡的想法

如果要把 Workflow 的精神搬到你的 bridge：

1. **動態 delegation plan** — 讓主 agent 在 goal 啟動時先輸出一個「編排計畫」（類似 dream.json 但是 per-task 動態生成），bridge parse 後按計畫 dispatch
2. **Cross-check pattern** — PARALLEL_DELEGATE 完成後，加一步把多個 specialist 的結果互相 review 再注入主 agent
3. **Script 持久化** — 成功的 delegation pattern 存成模板（像 workflow save），下次類似任務自動套用

不過你的架構已經比 Workflow 多了跨機器 + 持久記憶 + 使用者互動這些維度，兩者其實是互補的。

**Script 持久化 / workflow 存檔重跑**
