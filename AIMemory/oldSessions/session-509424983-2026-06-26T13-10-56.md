### ⚠️ 品質落差（可改進）

| 問題 | 具體表現 | 影響 |
|------|---------|------|
| **卡片深度不一** | `wild-transform`（6 個邊界案例、無 proto 欄位細節）vs `collect-feature`（9 個邊界案例 + 完整 proto + 4 個 enum） | 簡短卡片在實際翻譯時參考價值低 |
| **「規劃中」佔比高** | VS Feature、Feature Wheel、Global Multiplier、Progression Unlock、MAX WIN 的參考實作都標「規劃中」 | 沒有經過實作驗證的 pattern = 未驗假說 |
| **頻率欄位語意模糊** | 索引的「高/中/基礎」頻率沒定義什麼算高（3+專案出現？5+？） | 讀者無法判斷優先級 |
| **缺少「反模式」段** | 只列「怎麼做」，沒列「常犯的錯」或「不該這樣做」 | 防呆能力弱 |
| **SKILL.md 觸發條件過寬** | 「設計新 State / Manager 前查看」→ 實際上每次改 slot 都會被 skill router 命中 | 過度觸發 = noise |

### 🔧 建議改進項

1. **補齊薄弱卡片**：`wild-transform`、`mystery-symbol`、`buy-bonus` 對齊到 collect-feature 的深度（特別是 proto 結構和更多邊界案例）

2. **已驗證 vs 未驗證標記**：在索引加一欄 `驗證狀態`（✅ 已實作 / ⚠️ 僅規格推導），讓消費者知道可信度

3. **定義頻率門檻**：
   - 高 = ≥3 個專案用過
   - 中 = 1-2 個專案
   - 低/新 = 僅規格出現未實作

4. **增加「反模式」段**：每張卡加一小段 `## 常見錯誤`，例如：
   - Collect Feature：「把 Boost/Multi/Chain 四型誤植到 eye_strike（它沒有）」
   - Global Multiplier：「把格子乘倍和全局乘倍混用」

5. **觸發條件收窄**：SKILL.md 的觸發條件改為更精確的：
   - ~~「設計新 State / Manager 前」~~ → 「從規格書翻譯 Feature 為程式架構時」+ 「使用者明確問哪些模式可參考時」

6. **缺失 pattern 候選**（從已知專案推測）：
   - **Cascade/Tumble（消除補位）** — 你有 SlotReels.Explode+Fill 但沒獨立卡
   - **Free Game Retrigger** — scatter-collect 只涵蓋觸發，追加局數的決策邏輯沒獨立
   - **Gamble/DoubleUp** — 如果有的話
