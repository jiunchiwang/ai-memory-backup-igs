## 核心特色：Closed Learning Loop（閉環學習）

這是 Hermes 和所有其他 agent 框架的最大差異。五步循環：

1. **Receive** — 訊息從任何平台（Telegram / CLI / Discord…）進來
2. **Retrieve** — FTS5 全文搜索 persistent memory + skill 文件（~10ms / 10K+ 文件）
3. **Reason & Act** — LLM 推理 + 呼叫 tool
4. **Document** — 完成複雜任務（≥5 tool calls）後，**自動產出 skill 文件**
5. **Persist** — 結果、偏好、新知識存入持久記憶

**實測效果**：累積 20+ skill 後，同類型任務速度提升 ~40%（Nous Research benchmark）。

## 三層記憶系統

| 層級 | 用途 | 儲存 |
|------|------|------|
| Session memory | 當前對話 context | 記憶體 |
| Persistent memory | 跨 session 長期事實（FTS5） | SQLite `~/.hermes/state.db` |
| User model | 使用者偏好/風格/領域專長 | 自動累積 |

**限制**：MEMORY.md ~2,200 字、USER.md ~1,375 字（共約 20 條短記錄），超出後 agent 花 turn 做合併壓縮。透明度不如 OpenClaw 的檔案制。

⚠️ **關鍵陷阱**：`persistent_memory` 和 `skill_generation` **預設關閉**，需要手動在 config 啟用。

- `chat_completions`（OpenAI-compatible）

- **Cancel** — 支援取消正在進行的推理

## ACP Adapter（與你的 bridge 直接相關）

Hermes 自帶 ACP adapter（`hermes acp` / `hermes-acp`），是 JSON-RPC over stdio：

- `acp_adapter/server.py` → `HermesACPAgent` 實作 ACP 協議
- Session 管理、fork、cancel、model switching
- Permission bridge 把危險操作轉成 ACP permission request
- Tool rendering 把 patch/write_file 轉 diff 格式
- Session 持久化到 `~/.hermes/state.db`，重啟後可 resume

**意義**：你的 telegram-kiro-bridge 理論上可以用 `ACP_AGENT_COMMAND=hermes-acp` 直接接上 Hermes Agent，**零改 code**。

### 🤖 Assistant (下午10:46:40)

你是問我們的 bridge 用的 ACP 和 Hermes 的 ACP 差在哪吧？

核心差異不大——兩邊都是 **JSON-RPC over stdio**，遵循同一個 ACP 協議規範。主要差異：

## 協議層面（幾乎一樣）

兩邊都實作：
- `initialize` handshake
- `session/new`、`session/prompt`、`session/cancel`
- `session/update` 事件推送（streaming）

Hermes 額外支援：
- `session/fork`（深拷貝 session 歷史）
- `session/load` / `session/resume`（跨 restart 恢復）
- `session/list`（列所有 session）
- model switching（mid-session 換模型）

你的 bridge 的 `acpClient.ts` 比較精簡，只用 new + prompt + cancel + update 這四個核心 RPC。

## 架構層面（差異在這）

| | 你的 bridge（消費端） | Hermes ACP adapter（供給端） |
|--|--|--|
| 角色 | ACP **client** — spawn agent process、送 prompt、收 update | ACP **server** — 被 spawn、接 prompt、回 update |
| 語言 | TypeScript（Node.js） | Python |
| Session 持久化 | 不持久，idle timeout 就關 | SQLite `state.db`，重啟可 resume |
| Permission 機制 | 沒有（agent 自己管） | 有 approval bridge（危險操作攔截） |
| Tool 呈現 | 不管（透傳 agent 回覆） | 把 patch/write 轉成 diff 格式給 UI |
| 多 session | 每個 chatId 一個 session | 支援 fork + 並行 session |

## 實際影響

如果你想把 Hermes 接上你的 bridge：

```env
ACP_AGENT_COMMAND=hermes-acp
```

理論上直接能跑，因為 Hermes 的 ACP adapter 完整實作了你 `acpClient.ts` 用到的那幾個 RPC。**但**：

1. **Memory MCP 衝突** — Hermes 有自己的 memory 系統（SQLite），你的 `mcp-memory.ts` 會變成第二套記憶，兩邊不互通
2. **Skill 路徑** — Hermes skill 在 `~/.hermes/skills/`，你的 preamble 注入的 `SKILL_DIR` 指向 `~/.kiro/skills/`，要改 env
3. **Permission callback** — Hermes 會發 ACP permission request，你的 `acpClient.ts` 目前沒處理這個 event（會被忽略，等同 deny）

如果要正式支援，大概要：
- `acpClient.ts` 加一個 `permission/request` handler（轉成 Telegram ASK button 讓使用者批准）
- 決定 memory 歸屬（用 Hermes 的還是你的 MCP，或兩邊都開）

整體來說協議本身相容，差異在各自 adapter 的**功能豐富度**和**周邊生態整合**。
