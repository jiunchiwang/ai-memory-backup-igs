對，你的判斷是對的。

bridge 的 `default-skills/` 機制（postinstall 複製到各 CLI skill 目錄）是**舊設計**——在你建立 AI-canonical 正本 + junction 投影流程之前就存在的。現在既然：

1. `~/.kiro/skills/` 已經是 junction 指向正本
2. 你有 `sync.ps1 -Apply` 統一投影
3. steering/skill-workflow.md 明確規定「直接改 `~/.kiro/skills/` 會被覆蓋」

那**預設就是走新流程**沒錯。具體來說：

- **新增 skill** → 寫在 `G:\AI\AI-canonical\skills\general\<name>\SKILL.md`，跑 sync
- **修改 skill** → 直接改正本（junction 即時反映），commit + push
- **bridge 的 `default-skills/`** → 變成「給沒有 AI-canonical 的新機器 bootstrap 用」的 fallback，不再是日常編輯路徑

唯一要注意的是 bridge repo 的 git pre-commit hook 會把 `~/.kiro/skills/` 同步回 `default-skills/`——這跟 junction 不衝突，反而剛好：正本改了 → junction 反映 → commit bridge 時 hook 自動把最新版帶進 `default-skills/`，其他沒裝 AI-canonical 的機器 `npm install` 還是能拿到。

**結論**：預設走正本流程，`default-skills/` 是自動產生的投影副本，不是編輯入口。
