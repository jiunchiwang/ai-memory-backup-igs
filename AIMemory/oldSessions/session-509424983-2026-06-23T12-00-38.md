## Headroom 研究摘要

**Headroom** 是一個 **AI agent context 壓縮層**，47.2k stars，Apache 2.0，v0.27.0（2026-06-22 剛更新）。

### 核心定位

在 agent/LLM 收到 prompt 之前，壓縮 tool outputs、logs、RAG chunks、檔案、對話歷史——宣稱 **60–95% token 節省，答案品質不變**。

### 6 種壓縮演算法

| 引擎 | 對象 |
|------|------|
| **SmartCrusher** | JSON（陣列、巢狀物件） |
| **CodeCompressor** | AST-aware（Python/JS/Go/Rust/Java/C++） |
| **Kompress-v2-base** | HuggingFace 自訓模型，針對 agentic traces 文字 |
| **Image compression** | ML router，40–90% |
| **CacheAligner** | 穩定前綴讓 provider KV cache 命中 |
| **IntelligentContext** | 分數制 context fitting |

### 使用模式（4 種）

1. **Library** — `compress(messages)` Python/TS inline
2. **Proxy** — `headroom proxy --port 8787`，零改 code，任何語言
3. **Agent wrap** — `headroom wrap claude|codex|cursor|aider|copilot`
4. **MCP server** — `headroom_compress` / `headroom_retrieve` / `headroom_stats`

### 獨特功能

- **CCR（可逆壓縮）**：原文 cache 在本地，LLM 需要時可 `headroom_retrieve` 取回
- **Cross-agent memory**：跨 Claude/Codex/Gemini 共享 context，自動去重
- **`headroom learn`**：挖掘失敗 session，自動寫修正到 CLAUDE.md / AGENTS.md
- **Output token reduction**：不只壓輸入，也透過 verbosity steering + effort routing 減少模型輸出 token（Opus 輸出成本是輸入 5 倍）
- **Local-first**：資料不離開你的機器

### 技術棧

- Python 79% + Rust 16% + TypeScript 3%
- Rust crate 做效能敏感的壓縮核心
- Python 3.10+，uv 管理

### 與 bridge 的關聯性評估

| 面向 | 適用度 | 說明 |
|------|--------|------|
| **Preamble 壓縮** | ⭐⭐⭐ | bridge 的 preamble（facts + topic index + wiki + skill hints）很長，headroom proxy 可以在 ACP 送出前壓一輪 |
| **Tool output 壓縮** | ⭐⭐ | agent 的 tool call 回傳（file reads、grep 結果）可經 proxy 壓縮 |
| **Cross-agent memory** | ⭐ | bridge 已有自己的 memory 系統，重疊度高 |
| **headroom learn** | ⭐⭐⭐ | 自動從失敗 session 挖 lesson 寫入 AGENTS.md，跟 bridge 的 knowhow-accumulation / reflexion 很搭 |
| **Output shaping** | ⭐⭐ | 減少 agent 冗詞對 Telegram 訊息體驗有幫助 |

最簡單的方式是 **proxy mode**：
```

```
然後讓 ACP agent command 的 API base URL 指向 localhost:8787。但目前 bridge 用的是 **ACP stdio**（非 HTTP API），所以要嘛：

### 🤖 Assistant (上午11:27:35)

**ACP = Agent Client Protocol**

一句話：它是 **AI coding agent 的 LSP**——用 JSON-RPC 2.0 over stdio 標準化「編輯器/宿主 ↔ AI agent」的通訊。

### 重點

- **傳輸**：agent 從 stdin 讀、往 stdout 寫，每行一個 JSON-RPC message（`\n` 分隔）
- **設計理念**：類似 Language Server Protocol（LSP）但針對 AI agent 場景——session 管理、streaming 回覆、tool call、permission request 等
- **官網**：[agentclientprotocol.com](https://agentclientprotocol.com)
- **GitHub**：[agentclientprotocol/agent-client-protocol](https://github.com/agentclientprotocol/agent-client-protocol)

### 在 bridge 裡的角色

Bridge 的 `src/acpClient.ts` spawn 一個 agent process（如 `kiro-cli acp`），透過 stdio 跟它交換 JSON-RPC：
- `initialize` → handshake
- `session/new` → 開 session
- `session/prompt` → 送 user turn
- `session/update` → agent streaming chunk 回來
- `session/cancel` → 中斷

所以 bridge 本身是一個 **ACP client**，而 `kiro-cli acp` / `codex-acp` / `claude-agent-acp` 這些是 **ACP server（adapter）**。換 agent 只要換 adapter binary，協定層不變——這就是 bridge 能「改 `.env` 一行就切 agent」的原因。

## Headroom × Bridge 整合方案分析

研究完文件後，我整理出三個可行路徑，依**整合難度低→高**排列：

---

### 方案 A：MCP Server 掛給 Kiro agent（最容易，零改 bridge code）

**原理**：Headroom 提供 `headroom_compress` / `headroom_retrieve` / `headroom_stats` 三個 MCP tools。掛到 Kiro agent 後，agent 可以**自主決定**何時壓縮大量 tool output。

**步驟**：
```powershell
pip install "headroom-ai[mcp]"
headroom mcp install          # 自動寫入 ~/.kiro/settings/mcp.json 或 agents/main.json
```

**效果**：
- Agent 讀大檔 / grep 出大量結果時可呼叫 `headroom_compress` 壓縮後再推理
- 需要原文時呼叫 `headroom_retrieve`（1 小時 TTL）
- **不壓 preamble**（agent 看到 preamble 時 MCP 還沒介入）

**限制**：依賴 agent 自主判斷要不要壓；對 preamble/facts 注入無效。

---

### 方案 B：Proxy + `headroom wrap`（中等，需確認 Kiro CLI 支援）

**原理**：Headroom proxy 攔截 HTTP API call（`ANTHROPIC_BASE_URL`），壓縮 request body 裡的 messages。`headroom wrap claude` 會自動設 env 然後 spawn agent。

**問題**：Kiro CLI 走 ACP stdio，**agent 內部**呼叫 Anthropic API 的 base URL 是否可被外部 env 覆蓋？

- 如果 `kiro-cli acp` 會讀 `ANTHROPIC_BASE_URL` → 可以：
  ```powershell
  headroom proxy --port 8787
  # 修改 .env
  ACP_AGENT_COMMAND=kiro-cli acp
  # 在 start.bat 加：
  set ANTHROPIC_BASE_URL=http://127.0.0.1:8787
  ```
- 如果 Kiro CLI 不讀這個 env（很可能，它自己管 auth） → **此路不通**

**備案**：`codex-acp` / `claude-agent-acp` 可能比較容易，因為底層是 OpenAI SDK / Anthropic SDK 會尊重 `*_BASE_URL` 環境變數。

---

### 方案 C：Bridge 層 library 整合（最有效，改 bridge code）

**原理**：在 bridge 的 `buildPreamble()` 或 `session/prompt` 送出前，用 Headroom TS SDK 壓縮。

**架構**：
```
User message → bridge buildPreamble() 
                    ↓
            headroom compress(preamble + facts + tool outputs)
                    ↓
            ACP session/prompt（壓縮後的 content）
                    ↓
            Agent 看到精簡版 context
            （需要完整內容 → agent call headroom_retrieve MCP tool）
```

**實作**：
```typescript
// src/sessionManager.ts 或 src/memory.ts
import { compress } from 'headroom-ai';

// 在 buildPreamble 最後
const result = await compress(
  [{ role: 'system', content: fullPreamble }],
  { model: 'claude-sonnet-4-20250514', baseUrl: 'http://localhost:8787' }
);
const compressedPreamble = result.messages[0].content;
```

**前提**：必須跑 headroom proxy（TS SDK 需要 proxy 做後端壓縮）。

**風險**：
- 壓縮 preamble 可能丟失 bridge token 協定的關鍵指令（`<<ASK:...>>` 語法說明等）
- 需要把某些 section 標記為 `frozen`（不可壓縮）

---

### 方案 D：`headroom learn` 獨立工具（零整合，立即可用）

**原理**：用 Headroom 的 failure learning 掃 bridge 的 session logs，自動產出「agent 常犯的錯 + 修正方式」寫入 AGENTS.md。

**步驟**：
```powershell
pip install "headroom-ai[all]"
headroom learn --project G:\AI\telegram-kiro-bridge-main --apply
```

**效果**：
- 掃 Kiro session logs（`~/.kiro/` 下的 JSONL）
- 產出 path corrections、command patterns、search scope 寫入 AGENTS.md
- 跟 bridge 現有的 knowhow-accumulation / reflexion 互補

**限制**：需確認 `headroom learn` 的 Scanner 是否支援 Kiro log 格式（目前明確支援 Claude Code 的 `~/.claude/projects/*.jsonl`）。
