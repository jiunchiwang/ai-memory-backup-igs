## Loop Engineering 研究摘要

**cobusgreyling/loop-engineering**（⭐2.7k, MIT）是一個以「設計系統來 prompt agent，而不是自己去 prompt」為核心理念的參考 repo + CLI 工具集。

### 核心理念

> 「你不應該再自己 prompt coding agent。你應該設計讓系統替你 prompt agent 的 loop。」—— Boris Cherny（Anthropic Claude Code 負責人）

**Loop = 遞迴目標**：定義一個目的，讓 AI 搭配子 agent、驗證器、外部狀態反覆迭代，直到目標完成或升級給人。

---

### 六大構建積木

| Primitive | 作用 |
|-----------|------|
| **Scheduling** | 排程是心跳，沒排程就只是一次性 run |
| **Worktrees** | Git worktree 隔離，防止並行 agent 互踩 |
| **Skills** | 持久化意圖（專案慣例、踩坑紀錄），消除 intent debt |
| **MCP Connectors** | 讓 loop 能摸到 Jira/Slack/GitHub/DB |
| **Sub-agents (Maker/Checker)** | 寫的人不能改自己的作業——必須分開 |
| **Memory / State** | SESSION 外的持久脊柱（STATE.md 或 DB） |

---

### 7 個 Production Pattern

| Pattern | 節奏 | 風險 | 用途 |
|---------|------|------|------|
| Daily Triage | 1d–2h | Low | 每天早上自動整理 CI/Issue/PR 狀態 |
| PR Babysitter | 5–15m | Med | 監控 PR 卡住、提議修復 |
| CI Sweeper | 5–15m | Med | CI 紅燈自動分類+修 |
| Dependency Sweeper | 6h–1d | Med | 依賴升級，patch-only 先行 |
| Changelog Drafter | 1d | Low | 自動擬 release notes |
| Post-Merge Cleanup | 1d–6h | Low | 合併後清 dead code/TODO |
| Issue Triage | 2h–1d | Low | 自動分類標籤新 issue |

---

### 漸進式部署（L1→L2→L3）

- **L1（report-only）**：loop 只報告，不動作。第一週必須先跑 L1 確認品質。
- **L2（assisted fixes）**：loop 在 worktree 裡嘗試修復，需人 review。
- **L3（unattended）**：完全自主，有 allowlist + verifier 才放行。

---

### 10 大反模式

1. 同一個 agent 自己實作又自己驗證
2. 無重試上限（burn token）
3. Triage 產出模糊敘事而非結構化條目
4. 第一天就開 L3
5. 多 loop 共享無 schema 的 STATE.md
6. MCP 一開始就給 write-everything 權限
7. 無 kill switch
8. CI Sweeper 拿 code change 修 flake
9. 無 allowlist 的 auto-merge
10. 無 run-log（事後無法追蹤 loop 做了什麼）

---

### 關鍵概念詞彙

| 概念 | 意義 |
|------|------|
| **Intent Debt** | Agent 每次 cold start 缺少的意圖，靠 Skill 償還 |
| **Comprehension Debt** | loop 產出你沒讀的程式碼 → 理解落差 |
| **Cognitive Surrender** | 讓 loop 跑卻自己停止思考的陷阱 |
| **Orchestration Tax** | 平行 agent 的人類協調成本 |
| **Harness vs Loop** | Harness = 單次 session 環境；Loop = harness + 排程 + 狀態 + 驗證鏈 |
