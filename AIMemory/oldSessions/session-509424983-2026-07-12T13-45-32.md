## Multi-AI Chat Desktop — 快速摘要

**一句話**：Tauri 2 桌面 App，用一個控制面板同時操控已登入的 ChatGPT / Claude / Gemini / Grok **網頁版**，讓它們互相回答、辯論、交叉審查同一個問題。

### 核心設計

| 面向 | 做法 |
|------|------|
| **Runtime** | Tauri 2（Rust + WebView2/WebKit） |
| **前端** | TypeScript 86.7% + Vite + Tailwind |
| **LLM 接入** | **零 API key**——直接操控你已登入的各家 Web UI（WebView 注入 JS adapter） |
| **Adapter 模式** | `adapters/` 資料夾，每家一個 JS 注入腳本控制網頁 DOM/輸入/讀取回覆 |
| **隔離** | 各 provider 獨立 local WebView profile（cookie 不互串） |
| **跨模型協作** | 非 API 串接，而是「把 A 的回覆貼給 B 當 prompt」的 orchestration |

### 五種工作流模式

1. **Free** — 並行比較（brainstorm）
2. **Debate** — Pro → Con → Judge → Synthesis
3. **Consult** — 兩個獨立回答 → Review → Final
4. **Coding** — Spec → Review → v1 → Tests → v2 → Accept → Final
5. **Roundtable** — 5 rounds × 4 AI = 20 turns 對抗收斂

### 特點

- 純 local、無 telemetry、MIT 授權
- 支援 Windows / macOS (Apple Silicon) / Linux
- Session 30 個本地存檔、Snapshot / Replay
- 內建 Codex + Claude Code 的 repo skill（一鍵 dev 環境啟動）
- 有繁中/日/德翻譯
- 作者 Ted Huang（台灣），AI-Sister.com 贊助

### 狀態

- 最新 v1.0.1（2026-07-11）
- 作者宣布 **feature complete**，未來只修 adapter 相容性和安全

## 跟你的 telegram-kiro-bridge 比較

| | Multi-AI Chat Desktop | telegram-kiro-bridge |
|--|--|--|
| **介面** | 桌面 App（Tauri GUI） | Telegram Bot（行動/跨裝置） |
| **LLM 接入** | WebView 操控各家網頁（無 API key） | ACP protocol / Direct API（需 API key 或 CLI subscription） |
| **協作模式** | 多 AI 互相辯論/審查同一問題 | 單主 agent + specialist 分身 + relay 跨機 |
| **Tool / MCP** | 無（只有聊天） | 完整 MCP、tool call、檔案操作、記憶 |
| **長期記憶** | 無（local session transcript） | 完整 facts/topic shard/wiki 系統 |
| **適用場景** | 「同一個問題問四家比較」的決策輔助 | 「一個 agent 全自動幫你做事」的 autonomous workflow |
| **可擴展性** | adapter 注入腳本（脆，受 Web UI 改版影響） | ACP 標準協定（穩，換 agent 零改動） |

### 有趣的借鏡點

1. **Debate / Roundtable 模式**：多模型交叉質疑直到收斂——你的 `verifier.ts` + `self-fusion.ts` 有類似概念但是 programmatic 的，這裡是更 user-facing 的「看它們吵架」
2. **Zero API key 策略**：直接操控 Web UI 繞開 API 付費——聰明但脆弱（每次 UI 改版都要修 adapter）
3. **Repo Skill 標準**：同時放 `.agents/skills/` + `.claude/skills/`，跟你的 `default-skills/` 自動安裝概念類似

### 可能的啟發

- 如果想在 bridge 加一個「同時問 3 家模型再 synthesize」的 meta mode，邏輯其實跟 Consult mode 很像——你已經有 `PARALLEL_DELEGATE` + `self-fusion.ts`，差的是一個 user-facing `/compare` 指令
- 他的 adapter 注入方式（WebView + JS 操控 DOM）不適用於你的 server-side 架構，但「provider UI 改版風險」這個問題你用 ACP 標準協定完全避開了
