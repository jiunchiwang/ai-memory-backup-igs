## 懸賞令設資料時的進退場原理

### 整體概念

懸賞令（Wanted Poster）是遊戲畫面上方/側邊常駐的「追蹤卡片」，每張卡片綁定一個海盜符號（`BaseSymbol`，如 M1~M9）。每一輪 spin 後，伺服器會給一組新的懸賞令清單，程式要決定：**哪些卡片要播「飛進來」動畫（進場）、哪些要播「退出」動畫（退場）、哪些不動（持續）**。

### 兩條路線

程式用一個 flag `isNewRound` 區分：

**路線 A：新一局（RebuildAll）**
- 不管三七二十一，場上所有卡片全部退場（播 Out 動畫 → 關閉）
- 全部退完後，新清單的卡片依序進場（設資料 → 開啟 → 播 In 動畫）
- 白話：**洗牌重來，舊的全收、新的全擺**

**路線 B：同一局・消除後補位（ReconcileCascade）**
這是核心的「對帳」邏輯，用 multiset（計數比對）判斷每張舊卡片的命運：

| 情境 | 判定條件 | 處理 |
|------|---------|------|
| **持續** | 新舊清單都有同一個 BaseSymbol，倍率沒變，盤面上仍看得到此符號 | 什麼都不做 |
| **退場再進場** | 同一個 BaseSymbol 還在，但倍率變了（升級了）或盤面已看不到此符號 | 先播 Out → 換資料 → 再播 In |
| **純退場** | 舊清單有，新清單完全沒有這個 BaseSymbol 了 | 播 Out → 關閉 |
| **純進場** | 新清單多出一個 BaseSymbol，舊清單沒有 | 找空位（或等退場騰出位子）→ 設資料 → 開啟 → 播 In |

### 對帳的執行流程（ReconcileCascade）

```
1. 把新清單按 BaseSymbol 分好池子（pool）
2. 掃一輪舊綁定：
   - 空位 → 記住等新卡片用
   - 有綁定 → 去 pool 找配對：
     * 找到且「同倍率 + 盤面可見」→ 持續，從 pool 消耗一筆
     * 找到但「倍率不同 或 盤面看不到」→ 退場再進場
     * pool 裡完全沒有 → 純退場，騰出版位
3. pool 剩餘的 = 新進場的卡片：
   - 優先用現成空位立即進場
   - 不夠就等退場動畫播完，用騰出的位子再進場
```

### 為什麼要這麼複雜？

如果不做對帳，每次都全收全擺，畫面會一直閃爍。用 multiset 比對後：
- 沒變的卡片「靜靜待著」→ 視覺穩定
- 升級的卡片「退出再進入」→ 有明確的動畫反饋告訴玩家「倍率升了」
- 新增/移除的卡片各自有進退場動畫 → 有動態感但不混亂

### 觸發懸賞令的演出流程（WantedPosterState）

當消除的符號命中某張懸賞令的 BaseSymbol 時，會進入一段表演：

```
① Mark：常駐卡片閃一下「標記中」
② Symbol_In：場上消除位置 spawn 一個 Spine 特效飛入
③ Mark_Active ∥ Symbol_Active：常駐卡片 + 場上 Spine 同時高潮動畫
④ 底板升級：更新該位置的乘倍底板（帶動畫）
```

多組不同 BaseSymbol 同時被觸發時，是並行演出（`Promise.all`），不會一組一組排隊。簡單一句話總結：**懸賞令用「前後清單 BaseSymbol 計數比對」決定每張卡片命運——同的不動、新的飛進來、消失的飛出去、升級的退出再進入——避免全部重建造成畫面閃爍。**

## 問題根因

退場的 item `node.active = false` 後，cc.Layout 偵測到子節點變化，在**同一幀**重新計算剩餘 item 的位置 → 存活的卡片瞬間跳到新位置，沒有過渡。

---

## 解決方案

### 方案 A：退場完成後「先動畫滑到新位置，再讓 Layout 接管」

流程：
1. 退場 item 播完 Out 動畫後，**暫時不關 active、也不讓 Layout 重排**（先把 Layout.enabled 設 false 或把退場 item 的 `widget` 移出 Layout 容器）
2. 手動計算剩餘 item 的目標位置（或直接讓 Layout 空跑一次取得目標 pos）
3. 對存活 item 執行 `tween(node).to(0.2, { position: targetPos })` 滑到位
4. 滑完後才關掉退場 item 的 active、恢復 Layout.enabled

優點：最小侵入，只要在 `PlayOut().then()` 後面加一段補間
缺點：需要手算或暫時模擬 Layout 目標位置

---

### 方案 B：不用 cc.Layout，改手動管理位置（slot array）

流程：
- 每個 item 固定分配 index 位置（像格子），用 `positions[i] = baseOffset + i * spacing` 自算
- 進場：tween 從畫面外滑到目標位
- 退場：tween 滑出畫面外再 active=false
- 剩餘卡片要壓縮（居中）時：tween 到新 offset

優點：完全控制動畫時機，不會突跳
缺點：放棄 cc.Layout 的自動排版，直橫切換要自己算兩套位置

---

### 方案 C：退場 item 不關 active，改設 opacity=0 + 尺寸縮為 0（漸變讓位）

流程：
1. PlayOut 動畫最後一幀把 item 的 `UITransform.width/height` tween 到 0（或 scale 到 0）
2. Layout 會因為子節點尺寸變化**漸進式**推擠剩餘 item
3. 尺寸歸零後才 `active = false`

優點：最簡單，利用 Layout 的自然反應，讓位移是連續的
缺點：Layout 計算頻率是每幀，tween 期間 Layout 會每幀重排（效能微增，但懸賞令最多 2~3 張影響不大）

---

### 方案 D：延遲退場——讓退場 item 播「縮小 + 淡出」動畫，動畫本身就包含「讓位」

流程：
- PlayOut 改為「scale 0.3s 縮小到 0 + opacity fade」，不額外播 spine Out
- Layout 追隨 scale 變化自動推擠旁邊的 item
- 動畫結束 active=false

優點：退場視覺本身就包含空間釋放，不需要兩段動畫
缺點：可能和現有 spine Out 動畫風格不搭，需要美術確認

---

### 方案 E：Layout 加上 cc.Widget 的 transition 效果（hack 層）

在 Layout 容器與 item 之間插一層 wrapper Node：
- wrapper 的 position 由 Layout 控制
- item 是 wrapper 的子節點，用 `tween` 平滑追蹤 wrapper 位置變化

優點：Layout 邏輯完全不動，只多一層視覺緩衝
缺點：結構多一層，排查稍複雜

---

## 推薦

| 情境 | 推薦 |
|------|------|
| 想最小改動 + 保留現有 spine 動畫 | **方案 A** |
| 想最簡單實作且美術可接受 | **方案 C** |
| 之後還有更多動態排版需求（如 3→1 張） | **方案 B**（一勞永逸） |
