## 「訊息消失再出現」的根因分析

**Draft mode 路徑有 30 秒 TTL 黑洞：**

`trySendDraft()` 的 doc 明確寫了「Draft 是 30s TTL ephemeral」。streaming 過程中如果遇到 rate limit（`rateLimitedUntil() > Date.now()`），`editNow()` 直接 return，**不送任何 draft 更新**。但 draft TTL 照跑。

流程是這樣：
1. `trySendDraft()` 建立 draft → 使用者看到 "Thinking…"
2. Streaming 開始，`editNow()` 每隔 interval 更新 draft
3. 如果中途遇到 429 rate limit（`retry_after` 可能 30-180 秒）
4. `editNow()` 在 L446 檢測到 rate limit → **直接 return，不更新 draft**
5. **30 秒後 draft TTL 過期 → 訊息從使用者畫面消失**
6. Rate limit 結束 → `onRateLimitRecovery` 觸發 → `editNow()` 恢復
7. 但 draft 已過期 → `trySendDraft()` 可能靜默失敗（return false）→ fallback 到 placeholder，但 **placeholder 從未建立**（因為走了 draft 路徑）

**結果：rate limit 期間 draft 過期消失，recovery 後沒有 placeholder 可以退回，訊息就「不見了」直到 final delivery 用 `sendMessage` 才重新出現。**

### Placeholder 路徑也有類似問題（但較輕微）

Non-draft 路徑：rate limit 期間 `editNow()` return → 訊息不更新（卡在舊內容），但 placeholder 不會消失。使用者看到的是「卡住」而不是「消失」。recovery 後會補上最新內容。

### 修復方向

Draft mode 需要一個 **keep-alive 機制**：rate limit 期間仍要定時送空的 draft 更新防止 TTL 過期，或在 recovery 時檢查 draft 是否已過期、改走 `sendMessage` 重建可見訊息。
