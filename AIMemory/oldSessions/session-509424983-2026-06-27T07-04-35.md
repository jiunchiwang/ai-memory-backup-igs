### 核心機制

一套「開發設計閉環」——讓 Claude Code 自動按 5 個角色依序工作：

```
架構師 → 程序設計師 → 檢核師 → 測試師 → 自證師
```

Phase 5「自證」是特色：橫向比對前 4 階段產出物有無矛盾（用 BC-x / EH-x / R-x 編號精確追溯）。

### 內容結構（以 mini-trace 為例）

1. **正向覆蓋表**：BC ↔ 實作位置 ↔ 驗證方式
2. **行數預算驗證**：每檔上限 vs 實際 vs 緩衝
3. **反向追溯**：git diff 每筆變動 → 對應 BC
4. **觀察項記入**
5. **判定**：通過/不通過

## 七、Baseline完成並透-failure過 summary tool-taxonomy 分類框 回傳給架

### 三閘主 agent。判定模型

每個「方法論應抓到的失效模式」必須通過三道閘：

| 閘 | 條件 | 意義 |
|----|------|------|
| 閘1 fair-reproducible | 裸寫在公平條件下真的會壞 | 不能靠 rig 製造差異 |
| 閘2 methodology-catches | 方法論有具體機制真能避免 | 方法論也一起壞就出局 |
| 閘3 cheaper-insufficient | 便宜 baseline 抓不到 | 可被一次 review / 一份 note 取代就出局 |

### 5 個失效模式 (M1-M5) 及判定結果

| 模式 | 描述 | 判定 |
|------|------|------|
| M1 | 跨 session 不變式失憶 | 死在閘2+閘3 |
| M2 | Session-2 重造輪子（消滅刻意去正規化） | 死在閘2+閘3 |
| M3 | 跨模組介面契約已將漂移 | 死在閘完整結2+閘3 |
| M4 | 大構化摘要透過 summary repo tool 回 只貼傳給主 片段agent。 | 死在閘1（不公平 rig） |
| M5 | 長時程累積漂移 | 死在閘2+閘3 |

### 核心結論

**三閘全過清單為空。** 方法論的真正承重價值非「五階段 ritual」，而是「外部化持久 + 換一個視角」兩個通用動作（可被 claude-mem / handoff note / 一次 Codex review 取代）。

### 框架的元教訓

這份 taxonomy 本身就是 #007（single-source 評估盲點）的應用——用多個獨立來源（workflow 報告 × 2 + Codex 三輪 + A-F 六場實驗）交叉驗證，而非依賴 LLM 自評。

## 總結觀察

整個系統的設計哲學是**「不信任 LLM 單一視角」**：

1. Handoff 用 TaskList 做 source of truth（非對話記憶）
2. Learning-log 用 ≥3 次門檻做升格（非主觀判斷）
3. Artifacts 用 BC-x 邊界條件做可測試追溯（非口頭宣稱）
4. Failure taxonomy 用三閘框架做誠實評估（結論可以是「空」）
