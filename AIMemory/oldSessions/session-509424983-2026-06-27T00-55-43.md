### 改善方案（3 層，可獨立漸進實作）

**方案 A：Optimistic Cancel（最小改動，效果最大）**

核心思路：cancel 後 **bridge 側立即** 視為 turn 結束，不等 CLI 回應。

```typescript
// sessionManager.ts cancel() 改動概念
cancel(chatId: number) {
  const s = this.sessions.get(chatId);
  if (!s) return;
  s.client.cancel();
  
  // ✅ 新增：立即清 inflight，讓使用者可以馬上送新訊息
  s.inflight = false;
  s._toolInFlight = false;
  s.onUpdate = undefined; // 停止 streaming edit
  
  // 標記此 turn 為 cancelled，prompt() resolve 後 finally 不重複處理
  s._cancelledTurn = true;
  
  // 還是保留 force-kill 兜底，但降到 15s
  if (!s.client.isClosed) {
    const timer = setTimeout(async () => {
      if (s.client.isClosed) return;
      await s.client.close();
    }, 15_000);
    timer.unref?.();
  }
}
```

效果：
- 使用者按 `/cancel` → 立刻可以送新訊息（不再被 inflight 擋）
- streaming 編輯立即停止
- CLI 端 cancel 是否成功變成背景事件

風險點：如果 prompt() 在 cancel 後仍 resolve 了一段回覆，需要在 `finally` block 裡檢查 `_cancelledTurn` flag，跳過後續 token parsing / continuation / 等邏輯。

**方案 C：Aggressive Cancel（針對 tool 卡住場景）**

如果 `_toolInFlight === true`（CLI 正在跑 tool），cancel 後降為 10s timeout 就 kill process tree，因為：
- tool 正在跑時 CLI 端收到 cancel 通常也做不了什麼（shell 子進程可能無法 interrupt）
- 殺掉後用 `initializeAndCreateSession` 建新 session 恢復

```typescript
if (s._toolInFlight) {
  // Tool 卡住更激進：10s 直接 kill
  setTimeout(() => s.client.close(), 10_000);
}
```
