## Rich Messages Draft Streaming 功能說明

### 問題：舊做法

舊的 streaming 路徑是：
1. 先發一則 placeholder 訊息（`"⏳ ..."`）
2. Agent 每產生新內容就 `editMessageText` 更新那則訊息
3. 完成後再 `editMessageText` 一次寫入最終內容

**痛點**：頻繁 edit 容易觸發 Telegram 429 rate limit（`retry_after` 愈疊愈長，60s→183s），半截 token 洩漏，UX 閃爍。

---

### 新做法：三階段 Draft Lifecycle

使用 Bot API 10.1 的 Draft API，streaming 變成三階段：

```
使用者: 「幫我分析這段程式碼」

[階段 1] 開始 — Thinking 動畫
┌───────────────────────────────────┐
│  💬 Thinking…                     │  ← sendMessageDraft(chatId, draftId, "")
│  （ephemeral，使用者看到打字動畫）   │     空 text = 自動顯示 "Thinking…"
└───────────────────────────────────┘

[階段 2] Streaming — 即時更新（每幀全量替換）
┌───────────────────────────────────┐
│  這段程式碼的主要問題是…            │  ← sendRichMessageDraft(chatId, draftId, content)
│  1. 變數命名不一致                  │     同 draftId 重複呼叫 = 全量替換 + 動畫過渡
│  2. 缺少錯誤處理…█                 │     Draft 是 ephemeral（30s TTL），不計入訊息歷史
└───────────────────────────────────┘

[階段 3] 完成 — 持久化
┌───────────────────────────────────┐
│  這段程式碼的主要問題是…            │  ← sendRichMessage(chatId, finalContent,
│  1. 變數命名不一致                  │       { reply_markup, reply_parameters })
│  2. 缺少錯誤處理                   │     Draft 自動消失，變成真正的訊息
│  3. 沒有 input validation          │     ASK 按鈕此時才掛上
│                                   │
│  [修復方案 A]  [修復方案 B]         │  ← reply_markup（inline keyboard）
└───────────────────────────────────┘
```

### 核心差異

| | 舊路徑（placeholder + edit） | 新路徑（Draft API） |
|---|---|---|
| 初始訊息 | `sendMessage` 真實訊息 | `sendMessageDraft` ephemeral |
| 更新方式 | `editMessageText`（會觸發 rate limit） | `sendRichMessageDraft`（draft 更新壓力較低） |
| 429 風險 | 高（每幀一次 edit） | 低（draft 更新機制不同） |
| UX | 閃爍、半截內容 | 原生 Thinking 動畫 + 平滑過渡 |
| 持久化 | 最後一次 edit 就是最終版 | 明確 `sendRichMessage` 持久化 |

### Fallback 機制

Draft 不是萬用的，自動降級：

```
私人聊天 + Draft 啟用  → ✅ Draft path
群組聊天              → ❌ 走舊 placeholder+edit（draft 不支援 group）
Relay 轉發            → ❌ 走舊路徑
Draft 某幀失敗        → 該幀退回 editMessageText
TG_DRAFT_ENABLED=0    → 完全回到舊行為
```

### 429 Recovery

Draft 有 30s TTL，rate limit 期間可能過期。但設計上不需特殊處理：

```
正常 streaming     → draft 更新
429 rate limit     → editNow() early return（現有邏輯不變）
Recovery 回來      → editNow() 推最新 buffer
                   → trySendDraft(同 draftId) = 自動重建 draft ✅
```

### 設定

`.env` 加一行即可控制：
```
TG_DRAFT_ENABLED=1   # 預設開（1/true/yes）
TG_DRAFT_ENABLED=0   # 關閉，完全回到舊行為
```

簡言之：**使用者看到的是原生 "Thinking…" 動畫 → 內容平滑出現 → 最終版帶按鈕**，不再是閃爍的 edit 連發。
