### 🤖 Assistant (上午9:05:36)

在 telegram-kiro-bridge 的語境裡，**preamble** 是每次建立新 ACP session 時，bridge 自動組裝並注入給 agent 的「系統前綴訊息」。

它包含：

1. **Environment block** — 機器路徑變數（`MEMORY_DIR`、`SKILL_DIR`、`AGENT_CONFIG_DIR`、`USER_ID`）
2. **長期記憶** — 最近 N 條 facts（由 `MEMORY_PREAMBLE_TAIL` 控制）
3. **Topic index** — 告訴 agent 有哪些 topic shard 可查
4. **Wiki index** — 可用的知識庫頁面
5. **Tool protocol** — bridge 支援的 token 語法說明（`<<SEND_FILE>>`、`<<ASK>>`、`<<SCHEDULE>>` 等）
6. **Skill routing hint** — embedding 比對後建議 agent 參考哪些 skill
7. **Memory recall** — 語意召回的相關 facts
8. **Daily log** — 當天的工作紀錄
9. **Working state** — 上次 session 的斷點狀態
10. **各種 policy** — reflexion、gap analysis、remember guidelines 等指引

關鍵設計原則（見 `POLICIES/preamble-frozen-snapshot.md`）：**每個 session 只 build 一次並凍結**，mid-session 不重建。這確保 agent 整個對話期間的行為基礎一致。

簡單說：preamble = agent 的「開場背景包」，讓它一啟動就知道你是誰、有什麼記憶、能用什麼工具、該遵守什麼規則。

### 更新策略（關鍵）

## 我的建議：B + C 混合（不需要開發新指令）

因為：

1. **建 AI.md 需要智能判斷**，不是跑個 script 就行 — 要讀 code、理解架構、判斷哪些是特有機制。一個命令能做的頂多是「觸發我去做這件事」，本質上跟直接跟我說沒差。

2. **Skill 規則已經寫了「首次進入新專案 → 若無 AI.md → 研究後建立」**。所以下次你叫我改 uk_722 的 bug，我進去發現沒有 AI.md，就會先建。

3. **如果你想更主動**，直接說：
   - 「幫 uk_722 建 AI.md」 → 我只建索引層（快，~2 分鐘）

4. **modules.md 不需要首次就建齊**。skill 的規則是「改到第二個模組時建立」— 因為你不確定會頻繁改哪些模組，等實際需要再補那一段最划算。

- 沒改過的專案建了也會過時
- 等你真的進去某個專案改東西時再建，反而是最新的
