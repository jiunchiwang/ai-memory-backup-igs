
## Hermes GPT v0.3.0 — 透過 MCP 接本地模型、繞過 Codex 配額

**發布日**：2026/06/27（昨天）

### 核心賣點

Hermes GPT v0.3.0 新增 **MCP 整合**，讓 ChatGPT 介面可以透過 MCP 協定把程式碼生成請求**路由到本機的 Hermes 模型**，不走 OpenAI Codex API → 無配額限制。

### 運作原理

```
使用者（ChatGPT 介面）→ MCP 協定 → 本機 Hermes 模型 → 回覆
```

- 對使用者：介面幾乎沒變化（還是用 ChatGPT 的 UI）
- 對 OpenAI：請求流程不透明（不經過 Codex endpoint）
- 本質：**前端保留 ChatGPT 體驗，後端切到自己的 local LLM**

### 解決的問題

OpenAI Codex API 有月度用量上限，重度使用者會撞牆。Hermes GPT 繞過這個：本地模型只受硬體限制，無外部配額。

### 取捨（文章有提）

| 優勢 | 劣勢 |
|---|---|
| 無配額限制 | 模型能力通常不及 OpenAI 最新一代 |
| 程式碼不上傳雲端（隱私） | 硬體有限時推理速度可能慢 |
| 免費（除電費） | 需自行維護本地環境 |

### 大脈絡

這不是孤立案例——文章指出一個趨勢：**當主流 AI 的用量門檻成為瓶頸，工程師主動找開源替代 + 透過 MCP 這類開放協議無縫嵌入現有工作流程**。MCP 作為 AI 工具互通的事實標準，讓這種「換後端不換前端」的策略可行。

---

### 跟你的 Bridge 關聯性

這篇文章描述的 Hermes GPT 比較偏向「ChatGPT 的前端替身 + 本地模型」的定位，跟你之前研究的 **Hermes Agent**（完整的 self-improving agent 框架）其實是不同的產品層面：

你的 bridge 已經有 llama.cpp 本地 LLM offload 7 個 NLP 任務，概念上走更遠——不是單純「換後端」，而是「主模型 + 本地模型各司其職」的混合架構。
