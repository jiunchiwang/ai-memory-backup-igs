實際 API 設計比預想更簡單：

### Telegram Draft API 真實行為

| 方法 | 行為 |
|------|------|
| `sendRichMessageDraft(chat_id, draft_id, rich_message)` | 用**同一個 `draft_id`** 反覆呼叫 = **全量替換** + 動畫過渡 |
| `sendMessageDraft(chat_id, draft_id, text)` | 同上的 plain text 版；空 text → 顯示「Thinking…」|
| `sendRichMessage(chat_id, rich_message)` | 最終持久化為真正訊息（draft 自動消失）|

- Draft 是 ephemeral，**30 秒自動過期**
- 同 `draft_id` 的更新有**動畫過渡**
- 只能用在 **private chat**（group/relay 不行）

### 這代表什麼

**`@grammyjs/stream` plugin 的 append-only 是 plugin 設計選擇，不是 API 限制。** Raw API 每次呼叫都是全量替換，和 bridge 的 rebuild-replace 模型完美相容。

### Path A 設計（低風險，~200 行）

```
現有：ctx.reply("⏳") → editMessageText(placeholder, content) → editMessageText(placeholder, final)
Draft：sendDraft(draftId, "") → sendDraft(draftId, content) → sendRichMessage(final + reply_markup)
```

**改動檔案：**
1. `telegram-rich-renderer.ts` — 加 `sendDraft()` / `sendFinalMessage()` helper
2. `run-prompt.ts` — streaming 路徑換用 draft API（~150 行改動）
3. `bot-setup.ts` — draft 方法的 rate limit 處理

**Fallback chain：** draft 不可用 or group chat → 走現有 placeholder+edit（零改動）

**行為契約：**
- BC-1: Draft streaming 期間顯示最新內容
- BC-2: 最終 `sendRichMessage` 持久化訊息
- BC-3: ASK 按鈕出現在 final message
- BC-4: Token stripping 行為不變
- BC-5: Rate limit 期間 draft 可能過期（30s），recovery 需重新建 draft
- BC-6: Relay/group 走 fallback
- BC-7: SEND_FILE/STICKER/SCHEDULE token 在 final 正常觸發
