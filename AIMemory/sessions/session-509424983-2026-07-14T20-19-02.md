### 是什麼

**psmux** = **Windows 原生的 tmux**。用 Rust 從頭寫（不是 port/wrapper），直接走 Windows ConPTY，相容 tmux 命令語言和 `.tmux.conf`。

- ⭐ 2.9k stars · v3.3.6（2026-06-13）· MIT
- 語言：PowerShell 59% + Rust 39%
- 安裝：`winget install psmux` / `cargo install psmux` / scoop / choco
- 需求：Windows 10/11 + PowerShell 7+
- 自帶 `tmux`、`pmux`、`psmux` 三個 alias binary

### 核心賣點

| 能力 | 說明 |
|---|---|
| Session 持久化 | 關視窗/斷 RDP/登出 → process 不死，reattach 繼續 |
| 83 tmux commands | split-window / send-keys / capture-pane / pipe-pane / hooks 等全有 |
| `.tmux.conf` 直讀 | 現有 tmux 主題（Catppuccin/Dracula/Nord）可直接用 |
| 全滑鼠支援 | 包含 SSH over Windows OpenSSH |
| 可腳本化 | `send-keys` 餵輸入 + `capture-pane` 讀輸出 = 自動化互動式程式 |

### 跟你最相關的：Claude Code Agent Teams 整合

psmux 對 Claude Code 有一級支援：
1. 自動設定 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` + `--teammate-mode tmux`
2. Claude Code 在 psmux session 裡跑時，teammate agent **各自分到獨立 pane**（可見、可監控）
3. 用 `capture-pane` 可程式化讀取任何 agent pane 的輸出

⚠️ Opus 偏好走 worktree（隱形 in-process），需要靠 CLAUDE.md 指示才會走 visible pane。

### 對 bridge 的潛在價值

1. **Specialist subprocess 可視化**：目前 bridge 的 specialist 是背景 ACP subprocess，用 psmux 可以讓每個 specialist spawn 在獨立 pane，方便 debug/觀察
2. **Session 存活力**：bridge 跑在 psmux session 裡就不怕意外關視窗、RDP 斷線
3. **`pipe-pane` logging**：把 bridge 或 agent 的 stdout 自動導到 log 檔，比現在的 `start.bat` 更強
4. **自動化測試**：`send-keys` + `capture-pane` 可以寫整合測試腳本，餵 input 驗 output
5. **多 agent 並行觀察**：跑 parallel delegate 時各 specialist 各一 pane

### 限制/不適合的點

- 需要 **PowerShell 7+**（你機器已有）
- 它是 terminal multiplexer 不是 process manager — reboot 後 session 不會自動恢復（需 psmux-resurrect plugin）
- 跟你現有的 `start.bat` loop 有重疊（都是為了 session 存活），但 psmux 更強大

### 結論

1. `winget install psmux` → 把 bridge 丟進 psmux session 跑，取代 `start.bat` 的重啟 loop（改用 `remain-on-exit` + `respawn-pane`）
2. 開發時多 pane dashboard（bridge log / acp trace / specialist output）
3. 長期可能整合 specialist 各自 spawn 到 pane

好問題。psmux session 的生命週期：

**還活著（bridge 繼續跑）：**
- 關閉 terminal 視窗 ✅
- Ctrl-B d（detach）✅
- 斷 RDP / SSH ✅

**真正關閉的情況：**

| 動作 | 結果 |
|---|---|
| `psmux kill-session -t bridge` | 殺整個 session（含裡面的 process） |
| `psmux kill-server` | 殺 psmux server（所有 session 全死） |
| 機器重開（reboot） | server 不見，session 全沒（除非裝 psmux-resurrect plugin） |
| 在 pane 裡 Ctrl-C 把 node 停掉 | process 結束 → 因為 `remain-on-exit` pane 留著，3 秒後 `respawn-pane` 自動重跑 |
| `exit` 退出 pane 的 shell | 同上，respawn 會重開 |

所以對你的 bridge 場景：
- **日常**：detach 就好，bridge 無感
- **想徹底停**：`psmux kill-session -t bridge`
- **重開機後**：需要重新跑 `pwsh start-psmux.ps1`（或加 Task Scheduler 開機自動跑）
