# Handoff

## Goal
為 Cocos Creator spine-viewer 插件新增「批量掃描指定目錄下所有 Spine，輸出 Excel 記錄每個 Spine 的最大 Triangle 和 DrawCall」功能。

## Completed
- 加入 exceljs 4.4.0 依賴
- 在 package.json 註冊 menu 項「Panel → Spine Batch Scan → Excel」
- 實作 main.ts 的 batchScan()：選目錄 → 選儲存路徑 → fork worker → Electron BrowserWindow 顯示進度
- 實作 batch-worker.ts：獨立進程執行掃描，不阻塞 Cocos 主進程
- Triangle 計算改為全 keyframe 掃描（從 animation timelines 抽取所有 frame time）
- DrawCall 改為精確版（CPU 模擬 PolygonBatcher 合批規則：texture identity + blend mode 斷批）
- dummy texture 修正（加 getImage()、每個 atlas page 回傳獨立物件）
- 進度視窗：Electron BrowserWindow 顯示進度條 + 當前檔名 + 完成結果 + OK 按鈕
- 建立 pack.bat 打包腳本，產出 spine-viewer-release.zip（19.4MB）
- editor.d.ts 補上 Editor.Dialog 型別宣告
- 全程 tsc --noEmit 零錯誤、tsc -b build 成功

## Decisions
- 因為 Cocos 3.6 的 Editor.Message.send 只路由到 main.ts methods，所以把掃描邏輯從 panel 搬到 main.ts（排除 panel message 跨進程傳遞因為 3.6 不支援）
- 因為 main process 同步計算會卡死 UI，所以用 child_process.fork 獨立 worker 執行（排除 Web Worker 因為 Electron main process 不支援）
- 因為進度視窗需要即時更新且 Cocos panel message 路由不可靠，所以用 Electron BrowserWindow + IPC 直接控制（排除 Cocos Panel 因為 message dispatch 在 3.6 行為不一致）
- DrawCall 選 CPU 模擬 batcher 合批邏輯（排除 headless-gl 因為需要 native compile、排除 OffscreenCanvas 因為會卡主進程）
- Triangle 改全 keyframe 掃描（排除 5 點取樣因為會漏掉特定 frame 的 peak）

## Pitfalls
- Cocos Creator 3.6 的 Editor.Message.send 只送到 package.json messages 註冊的 main.ts methods，不會自動轉發到 panel methods——嘗試走 panel message 浪費了時間
- spine-webgl TextureAtlas 建 region 時呼叫 texture.getImage()，dummy texture 必須提供此方法
- dummy texture factory 若用同一個 object reference，DC 比對永遠相同——必須每個 page 回傳獨立物件
- state.update(t) 是增量式 delta time，要做絕對時間定位必須先 setToSetupPose + setAnimation 再 update(t)
