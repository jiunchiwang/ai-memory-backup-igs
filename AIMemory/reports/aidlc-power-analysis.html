什麼是 AI-DLC？

AI-DLC（AI-Driven Development Life Cycle）是一套打包成 Kiro IDE Power 的自適應軟體開發方法論。它根據專案複雜度、風險、既有 codebase 狀態，智慧判斷哪些開發流程步驟值得執行、哪些可以跳過。

核心理念

"The workflow adapts to the work, not the other way around."
工作流程適配工作本身，而非反過來。

Key Benefits

特性說明
Adaptive Execution只跑有價值的 stage，簡單改動不拖泥帶水
Quality Gates每個 major stage 有 explicit user approval
Complete Audit Trail所有 user input + AI decision 帶 timestamp 寫 audit.md
Risk-Based高風險專案自動走全套，低風險精簡處理
Zero Dependencies不需要 MCP server、不需要外部服務，純 methodology

🔵 INCEPTION — 規劃 & 架構

決定做什麼以及為什麼做

  • 必跑 Workspace Detection — 掃描專案結構、判斷 brownfield/greenfield
  • 條件 Reverse Engineering — 分析既有 codebase（僅 brownfield）
  • 必跑 Requirements Analysis — 需求分析（depth 自適應）
  • 條件 User Stories — 使用者故事 & personas（有 user-facing 功能時）
  • 必跑 Workflow Planning — 產出執行計畫 + Mermaid 視覺化
  • 條件 Application Design — 元件/服務架構設計
  • 條件 Units Generation — 拆分工作單元

🟢 CONSTRUCTION — 設計 & 實作

決定怎麼做— 每個 unit 獨立走完整 loop

  • per-unit Functional Design — 業務邏輯設計（technology-agnostic）
  • per-unit NFR Requirements — 非功能需求 + tech stack 選型
  • per-unit NFR Design — NFR patterns + logical components
  • per-unit Infrastructure Design — 映射到實際基礎設施
  • per-unit 必跑 Code Generation — 分兩步：Planning → Generation
  • 必跑 Build and Test — 所有 unit 完成後統一跑

🟡 OPERATIONS — 部署 & 監控

目前只是 placeholder，未來擴展用。Build & Test 在 CONSTRUCTION 階段處理。

自適應判斷機制

AI-DLC 根據四個因子判斷每個 stage 是否需要執行：

因子說明
User's stated intent使用者描述的清晰度與完整度
Existing codebase有無既有 code（brownfield vs greenfield）
Complexity & scope影響範圍、跨元件程度
Risk & impact風險等級、是否影響 production

Adaptive Depth（三級深度）

深度條件做法
Minimal簡單、清晰的需求只記錄 intent analysis
Standard一般複雜度收集 functional + non-functional 需求
Comprehensive高複雜度、高風險完整需求追溯 + traceability

實際跑出來的差異範例

🔧 Simple Bug Fix
Workspace Detection → Requirements Analysis (minimal) → Workflow Planning → Code Generation → Build and Test
✨ New Feature in Existing App
Workspace Detection → Reverse Engineering → Requirements Analysis (standard) → Workflow Planning → Functional Design → Code Generation → Build and Test
🏗️ Greenfield Application
Workspace Detection → Requirements Analysis (comprehensive) → User Stories → Application Design → Units Generation → Workflow Planning → [Per-unit: NFR Req → NFR Design → Infra Design → Code Gen] → Build and Test

強制規則（Non-negotiable）

規則內容
Content Validation寫檔前必須驗證（尤其 Mermaid syntax）
Structured Questions問問題必須用 A/B/C/D/E + [Answer]: 格式
Explicit Approval每個 major stage 結束必須等 user 確認
Audit Logging所有 user input 原文 + ISO timestamp 寫 audit.md
Checkbox TrackingPlan + Stage 雙層 checkbox 即時更新進度
No Emergent BehaviorConstruction 階段只用標準 2-option completion menu

面向AI-DLC PowerClosed-Loop System

複雜度判定 多因子智慧評估（risk / scope / codebase state） 行數 + 檔案數三級制（微小/中型/大型）

Audit 強制 audit.md（ISO timestamp + raw input） 無獨立 audit，靠 fact + session transcript

設計哲學差異

AI-DLC 更好的地方

  • Audit trail 完整性（每筆 raw input 原文保留）
  • Lazy-loading 策略省 context
  • Per-unit construction loop 適合大型拆分
  • Overconfidence prevention 專用規則
  • Structured question format 標準化互動

Closed-Loop 更好的地方

  • Reflexion + 熔斷機制（自我修正）
  • Push-back 義務（對使用者提出質疑）
  • 因果鏈分析（修改前評估影響）
  • 事實主張閘門（分 A/B 級證據）
  • 三套粒度選擇（不 over-engineer 小任務）
  • 自動執行（不用每步都問人）

可借鏡的設計模式

1. On-Demand Workflow Loading

steering 只放調度規則 + 判斷條件；detail 到了該 stage 才 readFile 載入。

  • 你的場景：closed-loop-system.md 全量注入 preamble 消耗 context；可考慮把 Phase detail（如 P1 架構師、P3 檢核師的完整規則）拆出去，只在任務等級匹配時才讀
  • 難度：低——Kiro steering 本就支援多檔

2. Structured Audit Trail

獨立 audit.md 記錄所有互動（ISO timestamp + raw user input + AI response + context）。

  • 你的場景：目前靠 session transcript + facts；如果想追蹤 agent 決策品質/回溯問題，可加 lightweight audit
  • 可行做法：bridge 的 event-log.jsonl 已有類似基礎，可擴展

3. Per-Unit Construction Loop

大型任務拆成 units，每個 unit 獨立走完整 design→code→test cycle。

  • 你的場景：closed-loop 的 P2 實作師目前是線性的；大型任務可引入 unit 概念讓每個模組各自閉環
  • 注意：bridge 的 /goal 已有 multi-turn 能力，可配合

4. Overconfidence Prevention 規則

專門一份檔案教 AI 不要過度自信（不確定就標記、不猜測就問）。

  • 你的場景：closed-loop 的事實主張閘門 + 質疑熔斷已涵蓋類似功能，但沒有明確的「overconfidence」用語引導
  • 可做：在 karpathy-guardrails.md 加一條「不確定時明確標記信心等級」

不建議直接採用的部分

部分原因
每步 explicit approval gate對自動化 /goal 是反模式；你需要的是 agent 自主執行 + 出問題才停

最終結論

AI-DLC Power 是一套過程導向的 waterfall-lite 方法論，適合團隊協作 + PM 監管的場景。 對你的單人自動化 bridge + headless agent 來說：

  • ✅ 借鏡 lazy-loading steering 思路省 context
  • ✅ 借鏡 per-unit loop 處理大型任務拆分
  • ✅ 借鏡 overconfidence prevention 概念
  • ⚠️ 不需要它的 approval gate 密度
  • ⚠️ 你的 reflexion + push-back + 因果鏈比它更進階
  • ❌ 不適合直接安裝使用（Kiro IDE Power 格式 ≠ CLI steering）
