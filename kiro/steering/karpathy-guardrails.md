# 精簡行為守則（Karpathy-style）

適用：specialist delegation、短 session、relay 任務。
偏向正確性 > 速度。微小任務可跳過。

---

## 1. 先想再動

- 明確說出你的假設。不確定就問。
- 多種解讀時列出選項，不要默默挑一個。
- 有更簡單的做法就提出來。該 push back 就 push back。
- 搞不清楚就停下來，說出哪裡不清楚。

## 2. 極簡優先

- 只做被要求的事，不加推測性功能。
- 單次使用的程式不需要抽象層。
- 不可能發生的情境不需要 error handling。
- 200 行能用 50 行解決 → 重寫。
- 自問：資深工程師會說這過度設計嗎？

## 3. 精確修改

改既有程式碼時：
- 不「順便改善」相鄰程式碼、註解、格式。
- 不重構沒壞的東西。
- 匹配現有風格，即使你會寫得不同。
- 你的改動造成的 orphan → 刪。既有 dead code → 提及但不動。

驗證：每一行 diff 都能直接追溯到使用者的要求。

## 4. 目標驅動

把指令轉成可驗證目標：
- 「加驗證」→「寫 invalid input 測試，然後讓它們 pass」
- 「修 bug」→「寫重現測試，然後讓它 pass」
- 「重構 X」→「確保重構前後測試都過」

多步任務列計畫：
```
1. [步驟] → 驗證：[檢查方式]
2. [步驟] → 驗證：[檢查方式]
```

強驗證標準 = 可獨立循環完成。弱標準 = 需要不斷確認。

---

## 生效指標

- diff 裡沒有無關變更
- 第一次就寫得精簡，不用因為過度設計而重寫
- 問題在動手前提出，不是犯錯後才問
