## 需求探索（Section 1b）

需求不夠具體時，主動從多角度分析，整理為選項讓用戶選擇：
- **範圍界定**：MVP vs 完整版的切分建議
- **替代方案**：2-3 種技術路線的取捨比較
- **邊界情境**：容易忽略的極端/並發/異常情境
- **架構影響**：對現有系統的衝擊範圍

## 完整閉環 Phase 細節

### Phase 1：架構師 📐

**步驟**：
1. 兩層教訓查詢（長期警惕模式必讀 + learning-log 補充）
2. 模組資產查詢（module-registry.md 存在時）
3. 架構體質拆解（涉及現有模組時）
4. 指令轉換（命令式 → 可驗證目標）
5. 產出 BC-x/EH-x/IF-x + 分層聲明 + 驗證層級標注
6. 設計自檢（7 項）
7. 持久化到 `.claude-loop/artifacts/P1-design-spec.md`
8. 閘門檢查

**設計自檢 7 項**：
① 更簡單的做法？② 影響不相關模組？③ 邊界條件覆蓋？④ 操作流程變複雜？⑤ 解法複雜度成正比？⑥ 架構地基穩？⑦ 過度設計？

**⛔ 閘門**：所有 BC-x 有明確行為描述 ∧ 驗證層級已標注 ∧ 分層聲明完成 → 進 Phase 1b

### Phase 1b：設計審查 🔬

- 獨立子 agent 審查（讀取 design-reviewer.md）
- DR-x high → 回 Phase 1 修正（最多 3 輪）
- arch-risk → 記錄不阻擋
- medium/low → 讓用戶決定
- 用戶說「跳過設計審查」→ 直接進 Phase 2

### Phase 2：程序設計師 💻

- 增量驗證：每完成一個檔案立即編譯檢查
- 設計文件同步：BC-x 預期行為改變 → 同步設計規格
- ⛔ 閘門：無語法錯誤 ∧ 設計項都有實作 → 進 Phase 3

### Phase 3：檢核師 🔍

- 品質審查（code-reviewer.md）
- 安全審查（按領域預設，security-reviewer.md）
- R-x 嚴重度：high→斷點 A | arch-risk→記錄 | medium→建議修 | low→合併摘要
- ⛔ 斷點 A：有 high → 回 P2 修正 → 重跑 P3（差分審查）
- 學習日誌：斷點觸發時記錄

### Phase 4：測試師 🧪

- 確認每個 `[testable]` BC-x/EH-x 有對應測試
- `[visual-only]`/`[framework-dependent]` 列出驗證方式
- ⛔ 斷點 B：失敗 → 回 P2 修正 → 重跑 P3+P4

### Phase 5：自證師 ✅

**Part AB — 雙向追溯**（verifier.md）
**Part C — 整體評估**：
- 產出物驗證（Glob 確認 artifacts）
- 跨 Phase 一致性
- 回退規則：設計-實作不一致→P2 | 測試不足→P4 | 檢核未修→P2

**通過後**：
1. 學習日誌
2. 升格檢查（≥ 3 次 + 用戶確認 → 寫入問題追蹤「長期警惕模式」）
3. commit

## 精簡閉環細節

**步驟 1 — 設計**：
- 指令轉換（命令式 → 可驗證目標）
- 簡要設計規格（目標 | 函式簽名與型別 | BC-x ≥ 1 | 分層聲明）
- 設計自檢 7 項
- 持久化到 `.claude-loop/artifacts/P1-design-spec.md`

**步驟 1b — 設計快審（單輪）**：有 high → 修正；無 high → 進步驟 2

**步驟 2 — 實作**：每檔完成立即編譯檢查

**步驟 3 — 品質審查**：有 R-x high → 回步驟 2 → 重跑步驟 3

**步驟 4 — 測試驗證**：全通過 → 進步驟 4.5

**步驟 4.5 — 迷你追溯**：
每個 BC-x/EH-x 附 ✅/❌ + 檔案:行號（實作）+ 測試名稱
⛔ 正向覆蓋有 ❌ → 禁止 commit

**通過後**：學習日誌 + 升格檢查 + commit

## 斷點熔斷（全域規則）

同一 Phase 斷點累計 3 次 → 暫停，追加學習日誌（標記 `[circuit-breaker]`），由用戶決定：
- 繼續嘗試修正
- 降級為精簡閉環
- 重新設計（回 Phase 1）

## 修改前守衛深度規則

因果鏈分析 6 條深度規則：
1. 追溯根因
2. 穿透呼叫鏈 A→B→C
3. 語意影響（簽章不變但語意變）
4. 狀態與時序（初始化順序/事件時機/快取策略）
5. 邊界條件（驗證規則改後上游極端值）
6. 呼叫者存在性（grep=0 ⛔ 禁修改）

場景分流：
- 閉環 Phase 2/4：可省略根因但仍輸出影響和決策
- 斷點回退/非閉環修改：須完整分析
- 修改錯誤處理/降級/fallback：必執行第 6 條

## 事實主張閘門詳細規則

**觸發場景**：
- 將推論寫入記錄（特別是專案事實）
- 向用戶輸出確定語氣結論（含環境事實）
- 作為後續行動的事實前提

**處置**：
- **強**（≥ 1 A 級 + 反例通過）→ 可確定
- **中**（僅 B 級但反例通過）→ 標注「推論」，用戶確認後才記錄
- **弱**（反例未通過或不足）→ 不可輸出為事實

---

## Push Back 第 5 條詳細協議

**觸發條件**（4 條全成立）：
- 用戶斷言「X 是/在 Y」（含隱式）
- 未附證據
- 作後續行動前提
- 若假代價非微小

**輸出格式**：
```
⚠️ 我需要先驗證這個前提
├─ 用戶斷言：[X 是 Y]
├─ 證據要求：A 級 ≥ 1 / B 級+反例通過 /「請補證據」
├─ 我能查到的：[自查結果]
├─ 反例檢查：若真→X / 若假→Y / 實際→Z
└─ 處置：補證據 / 確認等級 / OK 用原方案
```

**解除條件**：
- 補 A 級或 B 級+反例 → 立即解除
- 「中度推論可接受」→ 解除
- 「OK 用原方案」但不補證據 → 不解除，要 explicit 確認「承擔事實錯誤代價」

## 產出物格式要點

- **BC-x**：邊界條件（可驗證行為）
- **EH-x**：錯誤處理
- **IF-x**：介面契約（跨模組）
- **R-x**：審查發現（high/medium/low/arch-risk/by-design）
- **DR-x**：設計審查發現

驗證層級標注：`[testable]` | `[visual-only]` | `[framework-dependent]`

詳細模板見 `.claudedocs/standards/產出物格式.md`（進入對應 Phase 時讀取）。
