# Self-Eval Prompt Pattern

## Overview

強制 AI 在交件前自我打分，不滿門檻自己改。把「品質判斷」從人類移到 AI 端，人類只需驗收最終版。核心：量化標準 + 強制扣分 + 不過關自動重做。

## When to Use

- 任何「品質有差的事情」：寫信、報告、簡報、貼文、設計提案
- 設計 Custom GPT / Project Instructions 時的品質閘門
- 需要 AI 不只「做完」還要「做好」的場景

## 標準模板

```
完成 [任務] 後，請強制執行自評：

1. **自我打分**（每項 0-100，給出具體扣分原因）
   - 完成度：是否所有要求都做了？
   - 品質：以「主管會驗收的標準」打分
   - 一致性：是否前後不矛盾？

2. **找出最弱項**：哪一項分數最低？根本原因是什麼？

3. **判斷是否要重做**
   - 任一項 < 85 → 立刻自己改，改完再評一次
   - 全部 ≥ 85 → 交件

4. **輸出格式**
   - 先給最終版本
   - 再列「自評：[完成度] / [品質] / [一致性]」
   - 最後列「【我做了哪些自我改進】1. ... 2. ...」

不要省略自評環節，不要在自評說「都很好」— 必須具體扣分。
```

## 門檻設計

| 場景 | 建議門檻 | 理由 |
|------|---------|------|
| 對外正式文件 | ≥95 | 零容錯 |
| 內部報告 | ≥85 | 允許小瑕疵 |
| 草稿/探索 | ≥70 | 快速迭代優先 |
| 搭配雙 Skill 循環 | ≥95 | QA 角色會嚴格把關 |

## 自評維度客製化

根據任務類型替換維度：

**寫作類**：hook 力 / 具體性 / CTA 自然度
**技術類**：正確性 / 完整性 / 可執行性
**設計類**：美感 / 可用性 / 一致性
**分析類**：邏輯性 / 數據支撐 / 結論明確度

## 防作弊機制

AI 會傾向給自己高分。對策：

1. **禁止空話**：「不要說『都很好』— 必須具體扣分」
2. **強制找弱項**：「最弱項是什麼？根本原因？」
3. **要求改進記錄**：「列出你改了什麼」— 沒改就是沒自評
4. **設具體禁忌**：「是否用了我禁止的詞？」— 可機械檢查

## 三級輸出策略

```
任一項 < 85  → 自己改完再交件（使用者不看中間版本）
85-94        → 交件 + 標「【可選改進】1. ... 2. ...」
全部 ≥ 95   → 直接交，不顯示自評過程
```

## 與「繼續優化」的搭配

自評是單次閘門，「繼續優化」是多輪迭代。組合用法：

```
1. 先跑自評 → 過 85 門檻
2. 再跑「繼續檢視可優化點」→ 多輪打磨到收斂
```

## Common Mistakes

❌ 門檻設太低（60 分）→ 等於沒設
❌ 維度太抽象（「整體品質」）→ AI 無法具體扣分
❌ 不要求扣分原因 → AI 會全給 90+
❌ 自評放在輸出最前面 → 使用者要翻很久才看到成品
❌ 每次都顯示完整自評 → 收斂後改成「< 85 才顯示」省版面
